# 明察（MingCha）· 视频问答智能体 需求文档

> **项目名**：明察（MingCha）——取自「明察秋毫」，一词覆盖"看懂（SUMMARY）、看准（LOCATE/VISUAL_LOCATE）、看住（MODERATE）"三种能力。CLI：`mingcha`（短命令 `mc`）；代码目录：`E:\AIAgent\mingcha`。
>
> 基于 `claude-real-video`（crv）构建的一个 AI 智能体：输入**视频地址**（本地文件或互联网 URL）+ **自然语言要求**（可选附一张参考图），输出**针对该要求的答案**。多模型可插拔（Claude / GPT-5.5 / GLM-5.2 …，原生 HTTP）。详细设计见 [视频智能体-详细设计文档.md](./视频智能体-详细设计文档.md)。

---

## 1. 项目背景与目标

### 1.1 背景
`claude-real-video`（CLI 名 `crv`）目前是一个**视频预处理工具**：把视频转成"LLM 可读的文件夹"（`frames/` + `transcript.txt` + `MANIFEST.txt` + 可选 `grids/`）。它本身**不回答问题**——回答由读取这些产物的 LLM 完成。

本项目要在 crv 之上加一层**智能体编排（orchestration）**，让"输入 URL + 一句话要求 → 直接拿到答案"这条链路自动跑通。

### 1.2 目标（一句话）
> 用户给一个视频地址、一句话要求（**可选再附一张参考图片**），智能体自动完成"取视频 → 按要求抽取合适的帧与文本 → 调用多模态模型分析 → 返回带证据（时间戳/截图）的答案"。

### 1.3 四类典型问题（必须覆盖）
| 类别 | 示例 | 对答案的要求 |
|---|---|---|
| **理解型（SUMMARY）** | "总结这个视频讲了什么"、"这个视频详细讲解了什么" | 结构化摘要，覆盖全片脉络 |
| **定位型（LOCATE）** | "车牌为琼A·xxxxx 的车最早出现在哪个时间" | **精确时间戳** + 证据截图 |
| **审核型（MODERATE）** | "有没有裸露隐私部位的人物"、"有没有出现暴力画面" | **高召回**的判定 + 所有命中片段的时间戳 |
| **以图搜视频（VISUAL_LOCATE）** | 附一张图片 + "这张图/这个人/这个物体在视频里出现在哪" | **精确时间戳** + 匹配帧截图 + 相似度 |

> **VISUAL_LOCATE** 是 LOCATE 的近亲：查询从"文字描述"换成"一张参考图片"，但复用同一套时间戳地基（G1）与两阶段扫描（§6.2）。可选支持"最早出现 / 所有出现区间"两种返回口径。

## 2. 现状能力盘点（crv 能做什么、缺什么）

> 依据：`src/claude_real_video/core.py`、`cli.py`（已通读）。

### 2.1 crv 现有能力（可直接复用）
- **取视频**：`fetch_video()` — URL 走 `yt-dlp`，本地文件直接复制；支持 `--cookies` / `--cookies-from-browser`。
- **场景感知抽帧**：`extract_frames()` — 单趟 `ffmpeg select`，命中场景变化或密度下限，保证时间顺序。
- **像素级去重**：`dedup_frames()` — 16×16 RGB 真实像素差 + 滑动窗口，去掉近似重复帧。
- **转写**：字幕 sidecar → 内嵌字幕轨 → whisper，三级回退。
- **拼图**：`make_grids()` — 3×3 contact sheet，帮模型跟上运动与进展。
- **完整音轨**：`extract_full_audio()` — 供能听音频的模型使用。

### 2.2 关键缺口（本项目必须补齐）

| # | 缺口 | 现状根因（代码位置） | 影响的任务 |
|---|---|---|---|
| **G1** | **帧无时间戳** | `dedup_frames` 重命名为 `frame_NNN.jpg`，丢弃时间信息（`core.py:163-169`）；`extract_frames` 用 `raw_%05d.jpg` 纯序号（`core.py:98`） | **LOCATE 完全无法回答"哪个时间"** |
| **G2** | **采样策略与检测冲突** | 去重 `dedup_threshold=8` + 抽稀 `max_frames=150`（`core.py:150-161`）为"总结"设计，主动丢帧 | LOCATE / MODERATE **会漏掉一闪而过的目标** |
| **G3** | **无编排/无问答层** | crv 只产出文件，不调用任何 LLM | 全部任务——需要有东西"读产物 + 回答" |
| **G4** | **无意图区分** | `--why` 只是把意图写进 MANIFEST 给人读的提示，不改变管线行为（`core.py:385-390`） | 三类任务需要不同的抽帧密度与分析策略 |
| **G5** | **grids 标注的是文件名不是时间** | `make_grids` 用 `os.path.basename(f)` 当标签（`core.py:331`） | 多模态模型无法从拼图直接读出时间 |

## 4. 详细功能需求

### FR-1 输入处理
- **FR-1.1** 接受 `source`：`http(s)://` URL 或本地文件路径（绝对/相对）。
- **FR-1.2** 接受 `prompt`：任意自然语言要求（中英文）。
- **FR-1.3** **接受可选 `query_image`：一张参考图片（本地路径或 URL）**，用于"以图搜视频"（VISUAL_LOCATE）。图片可与 `prompt` 同时给（如附图 + "这个人出现在哪"），也可只给图片（默认语义="在视频里找这张图的内容，返回出现时间"）。校验：图片可读、为受支持格式（jpg/png/webp）。
- **FR-1.4** 可选透传：`--cookies` / `--cookies-from-browser`（登录站点，仅限本人授权使用）、`--lang`、模型选择等。
- **FR-1.5** 输入校验：URL 不可达 / 文件不存在 / 无视频流 / 图片不可读 → 明确报错，不静默失败（注意 crv 的 `_run` 用 `capture_output=True` 静默执行，`core.py:12-13`）。

### FR-2 意图分类（Intent Classifier）
- **FR-2.1** 把请求归类为 `SUMMARY` / `LOCATE` / `MODERATE` / `VISUAL_LOCATE` 之一（允许一次请求命中多类，如"总结 + 有没有敏感画面"）。
- **FR-2.2** **有 `query_image` 时默认进入 `VISUAL_LOCATE`**（图片存在是强信号）；若同时有 `prompt`，用 `prompt` 细化查询语义与返回口径（"最早出现" vs "所有出现区间" vs "只判断有没有"）。
- **FR-2.3** 对 LOCATE / MODERATE / VISUAL_LOCATE，抽取**目标实体**：文字型来自 `prompt`（如"琼A 开头的车牌""裸露隐私部位"）；VISUAL_LOCATE 的目标来自 `query_image`——可选先让多模态模型对参考图生成一段**文字描述**（如"戴红色帽子的中年男性""白色 SUV 侧面"）作为辅助检索线索，见 §6.6。
- **FR-2.4** 分类实现优先用一次轻量 LLM 调用（few-shot 分类），附规则兜底（有 query_image → VISUAL_LOCATE；关键词：什么时间/第几秒/最早出现 → LOCATE；有没有/是否包含/检测 → MODERATE；总结/讲了什么/概括 → SUMMARY）。
- **FR-2.5** 分类结果、抽取实体、以及 VISUAL_LOCATE 对参考图生成的文字描述，都要回显给用户（可解释、可纠正）。

### FR-3 管线配置（Pipeline Planner）
按意图切换 crv 参数（数值为初始建议，需实测调优）：

| 参数 | SUMMARY | LOCATE | MODERATE | VISUAL_LOCATE |
|---|---|---|---|---|
| `fps_floor`（每 N 秒至少一帧）| 1.0（默认）| 0.5 或更密 | 0.5 或更密 | 0.5 或更密 |
| `scene`（场景灵敏度）| 0.30 | 0.20（更敏感）| 0.20 | 0.20 |
| `dedup_threshold` | 8（默认）| 低（3-4）| **关或极低** | 低（3-4）|
| `max_frames`（封顶）| 150 | 高/分段无封顶 | **不封顶（分段处理）** | 高/分段无封顶 |
| 时间戳 | 需要 | **必须** | **必须** | **必须** |
| grids 标签 | 文件名即可 | **时间戳** | **时间戳** | **时间戳** |
| 音轨 | 可选 | 视目标而定 | 可选（暴力/尖叫等声学线索）| 否 |

- **FR-3.1** SUMMARY 直接用现有默认路径 + `--grid`。
- **FR-3.2** LOCATE / MODERATE / VISUAL_LOCATE 必须启用**密集采样 + 时间戳**，去重放宽或关闭（见 §6.2 的两阶段方案控制成本）。

### FR-4 帧-时间戳映射（对应 G1，**最高优先级**）
- **FR-4.1** 抽帧阶段为每张保留帧记录其在视频中的**精确时间**（秒，保留小数）。
- **FR-4.2** 产出 `timestamps.json`：`[{"frame": "frame_001.jpg", "t": 12.5, "hms": "00:00:12.500"}, ...]`。
- **FR-4.3** 时间戳必须**穿过去重与重命名**存活（当前 `dedup_frames` 的 rename 会切断映射，必须改造）。
- **FR-4.4** 技术方案见 §6.1。

### FR-5 多模态分析（Analyzer）
- **FR-5.1 SUMMARY**：把 grids（或抽稀帧）+ 转写喂给 Claude vision，产出结构化摘要（主题 / 分段脉络 / 关键结论）。等价于当前"人读 MANIFEST"的自动化。
- **FR-5.2 LOCATE**：**按时间顺序**扫描帧（或带时间戳标签的 grids），找到目标实体**首次出现**的帧，返回其时间戳；建议两阶段（粗扫定位区间 → 该区间密集重抽 → 精确到秒，见 §6.2）。
- **FR-5.3 MODERATE**：对**每一帧/每一格**做审核判定（高召回优先），汇总所有命中片段的时间区间；对敏感内容只做**客观是否存在 + 时间**的判定与告警，不生成、不描述细节（见 §8 合规）。
- **FR-5.4 VISUAL_LOCATE（以图搜视频）**：把**参考图片**与视频帧一起交给多模态模型，按时间顺序找出**与参考图内容匹配**的帧，返回首次出现时间（或全部出现区间）。匹配可"由粗到细"：先用轻量视觉相似度（如感知哈希/嵌入向量）快速筛掉明显不相关的帧，再让多模态模型对候选帧做语义确认与打分（见 §6.6）。返回**相似度/置信度**，并对"外观相似但非同一对象"给出谨慎措辞。
- **FR-5.5** 每个判定必须附**证据**：命中帧文件名 + 时间戳 + 模型置信度/理由（VISUAL_LOCATE 额外附相似度分与"参考图↔命中帧"对照）。

### FR-6 结果组装
- **FR-6.1** 统一返回结构（见 §7.2）：`answer`（自然语言）+ `evidence`（时间戳 + 截图路径）+ `intent` + `confidence` + `caveats`（局限说明）。
- **FR-6.2** LOCATE 找不到目标时，如实回答"在抽取的帧中未发现"，并说明**采样密度下限**（可能存在于未抽到的帧）——不能假装确定不存在。
- **FR-6.3** 保留 crv 的诚实原则：转写为何缺失、帧为何被丢，都要能追溯。

## 5. 非功能需求

- **NFR-1 诚实性**：绝不把"没抽到"说成"不存在"。所有否定结论必须带采样局限声明。
- **NFR-2 可追溯**：每个答案可回溯到具体帧文件 + 时间戳，用户能自行核对。
- **NFR-3 成本可控**：长视频 + 密集采样 + 逐帧多模态调用非常烧钱/耗时；必须有两阶段策略、分段并发、帧数上限护栏。
- **NFR-4 本地优先**：延续 crv "全部本地处理"的定位；上云仅在调用多模态 API 时发生，且需明确告知用户视频帧将被发送到模型服务商。
- **NFR-5 可解释**：意图分类、参数选择、判定理由都对用户透明可纠正。
- **NFR-6 幂等/覆盖**：延续 crv 重跑覆盖输出目录的行为；同一 (source, prompt) 可缓存复用。

## 6. 关键技术方案

### 6.1 帧-时间戳映射（解决 G1）
当前 `extract_frames`（`core.py:96-99`）：
```python
_run(["ffmpeg", "-i", video,
      "-vf", f"select='gt(scene,{scene})+not(mod(n,{every_n}))',scale=640:-1",
      "-fps_mode", "vfr", os.path.join(frames_dir, "raw_%05d.jpg"),
      "-hide_banner", "-loglevel", "error"])
```
**改造思路**：在 select 后接 `metadata=print` 把每张被选中帧的 `pts_time` 落盘：
```python
# 在 -vf 的 select 之后追加：  ...select='...',metadata=print:file=frames_meta.txt,scale=640:-1
# frames_meta.txt 每帧一段, 含 pts_time:NNN.NN → 按顺序对应 raw_00001.jpg, raw_00002.jpg ...
```
- 解析 `frames_meta.txt`，得到 `raw_NNNNN.jpg → t 秒` 的初始映射。
- **改造 `dedup_frames`**：让它在去重/抽稀/重命名的**每一步都携带时间戳**（用 `list[dict]` 记录 `{raw, t, kept}` 而非裸文件名），最终输出 `frame_NNN.jpg → t` 的 `timestamps.json`。
- 这是 LOCATE 的地基，**必须先做**。

### 6.2 定位型的两阶段扫描（解决 G2 的成本问题）
密集逐帧扫全片对长视频不现实。建议：
1. **粗扫（coarse）**：用较稀的采样（如每 2-3 秒一帧）+ 时间戳，让多模态模型快速判断目标"大致落在哪个时间区间"。
2. **精扫（fine）**：只对候选区间用 ffmpeg **重新密集抽帧**（如每 0.2-0.5 秒），再逐帧判定，精确到目标**首次出现**的秒。
- 好处：全片粗扫成本低，精度只在必要区间投入，最终时间戳精确。

### 6.3 审核型的高召回采样（解决 G2）
- MODERATE **关闭或极大放宽去重**（`dedup_threshold` 调到接近 0），避免把关键帧当重复丢掉。
- 帧数按视频时长**分段（chunk）处理**，段内并发调用多模态模型，规避单次上下文与 `max_frames` 封顶。
- 判定取"**任一帧命中即命中**"的高召回策略，宁可多报（附时间让人工复核）也不漏报。

### 6.4 grids 时间戳标签（解决 G5）
改造 `make_grids`（`core.py:331`）：标签从 `os.path.basename(f)` 改为 `timestamps.json` 里对应的 `hms`。这样多模态模型看拼图时能**直接读出并引用时间**，无需再回查映射。

### 6.5 多模态模型选择
- **画面分析**：Claude vision（最新 Claude 模型，支持读多张图/拼图）。
- **音频线索**（可选，暴力/尖叫等）：能听音频的模型 + `extract_full_audio` 产出的 `audio.m4a`。
- 智能体调用模型时，把"目标实体 + 时间戳标注的帧/拼图 + 相关转写片段"一起给，要求模型**只依据画面作答并引用帧号/时间**。

### 6.6 以图搜视频的匹配流程（解决 VISUAL_LOCATE）
参考图 `query_image` 与视频帧的匹配，建议"**三级由粗到细**"，兼顾召回与成本：

1. **预筛（cheap，纯本地）**：对参考图与每张保留帧算**轻量视觉相似度**——可复用 crv 已有的 16×16 RGB 像素思路（`dedup_frames` 的 `sig`/`pct_diff`，`core.py:120-128`）做初筛，或用感知哈希 / 小型图像嵌入向量（如 CLIP）算余弦相似度。按相似度排序，保留 Top-K 候选帧。
   - 好处：把"逐帧调多模态"降为"只对少量候选调"，成本大幅下降。
   - 注意：像素/哈希对**角度、光照、尺度变化**不鲁棒；CLIP 嵌入对"语义同一物体"更稳，是更推荐的初筛器（列为选型开放问题，见 §10）。
2. **语义确认（多模态）**：把**参考图 + 候选帧**成对（或参考图 + 候选帧拼图）交给 Claude vision，问"这一帧里是否出现了参考图中的**同一对象/人/场景**"，返回布尔 + 相似度分 + 理由。
3. **精扫定位（可选）**：对确认命中的最早候选，按 §6.2 在其邻近时间区间**密集重抽**，把首次出现精确到秒。

**辅助线索**：可选先让多模态模型给参考图生成一段**文字描述**（FR-2.3），把 VISUAL_LOCATE 退化为一次文字 LOCATE 作为交叉验证——两条路（视觉相似 + 文字描述）都指向同一时间段时置信度更高。

**歧义与措辞**：区分"**同一个体**"与"**同类外观**"（如"就是这辆车" vs "一辆长得像的车"）。当只能判断"相似"无法判断"同一"时，答案必须如实降级措辞并降低 confidence。

## 7. 数据流与输出格式

### 7.1 中间产物（crv-agent 工作目录）
```
<out_dir>/
  source.mp4
  query_image.jpg    参考图片副本 (VISUAL_LOCATE 时)     (新增)
  frames/            frame_001.jpg ...          (保留帧)
  frames_meta.txt    ffmpeg metadata 原始时间信息 (新增)
  timestamps.json    frame → 秒 → HH:MM:SS.mmm  (新增, FR-4)
  grids/             grid_01.jpg ... (标签=时间戳) (增强, 6.4)
  transcript.txt
  audio.m4a          (可选)
  MANIFEST.txt
  answer.json        最终答案 + 证据             (新增, 7.2)
```

### 7.2 最终返回结构（`answer.json`）
```json
{
  "intent": "LOCATE",
  "target": "车牌 琼A 开头的车辆",
```

## 8. 合规与内容安全（审核型专门约束）

- **C-1** MODERATE 面向敏感内容（裸露、暴力等）时，智能体只输出**客观的"是否存在 + 时间位置 + 置信度"**，用于内容审核/过滤等正当用途；**不生成、不细节化描述**任何露骨内容。
- **C-2** 涉及未成年人的判定要格外谨慎，命中即高优先级告警并建议人工复核，不做进一步细节输出。
- **C-3** 处理他人视频/需登录的内容时，`--cookies` 类能力仅限用户对**本人账号/有权访问的内容**使用（延续 crv 现有措辞）。
- **C-4** 上云告知：调用多模态 API 会把视频帧发送到模型服务商，需在文档/首次运行时明确提示。
- **C-5** 隐私：涉及人物/车牌等可识别信息的截图为敏感数据，输出目录应提示用户妥善保管、按需清理。

## 10. 风险与开放问题

- **R-1 成本与时长**：长视频的 LOCATE/MODERATE 逐帧多模态调用可能很贵很慢——两阶段与分段并发是必需，仍需实测 token/耗时预算。
- **R-2 小目标可辨识度**：640px 宽的帧（`core.py:97` 的 `scale=640:-1`）对远处车牌可能**分辨率不足**。LOCATE 涉及细小文字/车牌时，可能需要更高分辨率抽帧或对候选帧做局部放大/裁剪二次识别。
- **R-3 采样召回上限**：任何非逐帧方案都存在"目标恰好落在两帧之间"的漏检可能——必须靠 caveats 如实声明（NFR-1）。
- **R-4 意图歧义**：一句话可能同时含多意图或表述模糊，需要分类器的多标签支持 + 必要时向用户澄清。
- **R-5 以图搜的"相似≠同一"**：VISUAL_LOCATE 存在**外观相似但非同一对象**的误报（同款车、撞衫的人）；也存在**同一对象但角度/光照/遮挡变化大**导致的漏报。必须在措辞与 confidence 上如实反映（§6.6），高风险场景建议人工复核。
- **开放问题**：
  - 是否需要 OCR（车牌/文字类 LOCATE 可显著提升准确率与成本效率）？
  - VISUAL_LOCATE 的**初筛器选型**：像素/感知哈希（零依赖、但不鲁棒）vs CLIP 等图像嵌入（更稳、需引入模型依赖）——权衡准确率、依赖体积与本地化。
  - 参考图里有多个对象时，如何指定"找哪个"（需要 prompt 配合，或让用户框选）？
  - 多模态模型的具体选型与配额？
  - 对外形态：CLI 子命令（`crv ask <url> "<prompt>" [--image ref.jpg]`）还是独立服务/API？
