## 0. 关键技术决策（先定，避免全文悬而未决）

需求文档 §10 留了几个开放问题，这里逐个拍板并给理由。若你不同意，改这一节即可，后文引用处会连带调整。

| 决策点 | 结论 | 理由 |
|---|---|---|
| **HTTP 方式** | **原生 HTTP（`requests`/`httpx`）**，不用官方 `anthropic` SDK | 需支持多家模型（Claude / GLM / GPT），统一走原生 HTTP + 自研 provider 适配层，避免被单一 SDK 绑定。 |
| **多模型架构** | **Provider 适配层**：抽象 `LLMProvider` 接口，两个实现——`AnthropicProvider`（Messages API 线格式）+ `OpenAICompatProvider`（Chat Completions 线格式，覆盖 GPT-5.5 / GLM-5.2 及一切 OpenAI 兼容端点） | 业界视觉+对话 API 实际只有两种线格式；两个适配器即可覆盖 Claude、OpenAI、智谱 GLM、以及大多数国产/开源兼容网关。**新增模型 = 加一行配置，通常不写代码。** |
| **画面分析模型** | 可配置，默认 `claude-opus-4-8`；备选 `gpt-5.5`、`glm-5.2`（经 OpenAI 兼容端点） | 需 vision + 长上下文 + 结构化输出。默认 Claude，用户可在 config/YAML 或 `--vision-model` 切换。**具体 model-id / base-url / 能力以各家官方文档为准**（见 §6.6 provider 注册表）。 |
| **意图分类模型** | 可配置，默认走"轻量档"（如 `claude-haiku-4-5` / `gpt-5.5-mini` / `glm-5.2-flash` 之一） | 分类是纯文本轻量任务（FR-2.3），用各家便宜档；模型池与画面分析解耦。 |
| **结构化输出** | **自研 `structured()`**：优先用各 provider 的 JSON-schema / json_object 模式；统一兜底为"prompt 里内联 schema + 解析首个 JSON + Pydantic 校验 + 失败重试 1 次" | 不再依赖 SDK 的 `messages.parse()`。不同 provider 的结构化输出能力不一，用"能力探测 + 统一兜底"保证任意 provider 都能拿到合法结构。 |
| **成本控制主手段** | ① prompt caching（**能力可选**，仅在 provider 支持时启用）② 批处理（**能力可选**，Anthropic Batches / OpenAI Batch）③ 两阶段扫描（§6.2，**provider 无关，始终生效**） | caching/batch 各家支持不一，做成"provider 能力开关"；两阶段扫描是纯算法，任何模型都受益，是**兜底的省钱手段**。 |
| **VISUAL_LOCATE 初筛器** | P2.5 先用**复用 crv 的 16×16 RGB 像素相似度**（零依赖）；CLIP 作为可选插件后置 | 需求文档 §6.6 明确"可复用 `dedup_frames` 的 sig/pct_diff"；先零依赖跑通，CLIP 引入 torch 依赖重，留作 R-5 优化。与 provider 无关。 |
| **OCR** | 不进 MVP；作为 LOCATE 车牌/文字类的**可选二次识别**插件（P2 之后） | 需求文档 R-2/开放问题；先靠 vision 高分辨率抽帧解决，OCR 是精度增强而非必需。 |
| **对外形态** | ① 新增 CLI `mingcha ask`（短命令 `mc ask`；含 `--vision-model`/`--provider` 等）；② 暴露 Python API `mingcha.ask(...)` | 需求文档开放问题；独立 CLI 便于分发，Python API 便于集成。**不引入独立服务**（超出 open-core 定位）。 |
| **默认输出** | 沿用 crv 覆盖式输出目录 + 新增 `answer.json`（§7.2） | 延续 NFR-6。 |

> ⚠️ **关于 GLM-5.2 / GPT-5.5 的免责声明**：这两个型号是用户指定的目标模型，本设计**不硬编码**其确切 endpoint、model-id、上下文窗口、是否支持 vision/结构化输出/批处理等——这些必须以各家**官方文档为准**。设计把它们全部做成 **config 驱动**（§6.6 的 provider 注册表），实现时按官方文档填表即可，无需改代码。若某型号不支持 vision，则它只能承担意图分类（纯文本）角色，画面分析回退到支持 vision 的 provider。

## 2. 数据结构（`mingcha/types.py`）

统一用 Pydantic（供 §6.4 的跨 provider 结构化输出直接校验 LLM 返回）+ 少量 dataclass（内部传递）。

```python
from enum import Enum
from pydantic import BaseModel, Field

class Intent(str, Enum):
    SUMMARY = "SUMMARY"
    LOCATE = "LOCATE"
    MODERATE = "MODERATE"
    VISUAL_LOCATE = "VISUAL_LOCATE"

# ---- FR-2 意图分类的 LLM 结构化输出 ----
class IntentResult(BaseModel):
    intents: list[Intent]                 # 允许多标签 (FR-2.1)
    target: str | None = Field(None, description="检测/定位目标实体, 文字型来自 prompt")
    return_scope: str = Field("earliest", description="earliest | all_ranges | exists")
    reason: str                           # 可解释 (FR-2.5)

# ---- FR-4 时间戳地基 (G1) ----
class FrameStamp(BaseModel):
    frame: str        # "frame_001.jpg"
    t: float          # 秒, 保留小数
    hms: str          # "00:00:12.500"

# ---- FR-3 管线配置 ----
class Plan(BaseModel):
    intents: list[Intent]
    fps_floor: float
    scene: float
    dedup_threshold: float
    max_frames: int | None      # None = 不封顶(分段)
    need_timestamps: bool
    grid_label_time: bool
    keep_audio: bool

# ---- FR-5/FR-6 证据与答案 ----
class Evidence(BaseModel):
    frame: str
    t: float
    hms: str
    confidence: float
    similarity: float | None = None   # 仅 VISUAL_LOCATE
    note: str = ""

class Answer(BaseModel):
    intent: str                    # 主意图(多意图时取首)或 "SUMMARY+MODERATE"
    target: str | None = None
    query_image: str | None = None
    answer: str
    evidence: list[Evidence] = []
    confidence: float = 0.0
    caveats: str = ""              # NFR-1 采样局限, 必填非空当结论为否定
    artifacts_dir: str
```

**`answer.json`** 就是 `Answer.model_dump_json()`，字段与需求 §7.2 逐一对应。

## 3. G1 时间戳地基（`core.py` 改造 + `timestamps.py`）—— 最高优先级

需求文档 FR-4 / §6.1。这是所有定位类（LOCATE/VISUAL_LOCATE/MODERATE）的地基，**P0 先做**。

### 3.1 `extract_frames` 改造：落盘 pts_time

当前（`core.py:96-99`）无时间信息。改为在 select 后接 `metadata=print`：

```python
def extract_frames(video, frames_dir, scene, fps_floor):
    os.makedirs(frames_dir, exist_ok=True)
    every_n = max(1, round(_fps(video) * fps_floor))
    meta_path = os.path.join(frames_dir, "frames_meta.txt")
    _run(["ffmpeg", "-i", video,
          "-vf", (f"select='gt(scene,{scene})+not(mod(n,{every_n}))',"
                  f"metadata=print:file={meta_path},"          # ← 新增: 逐帧打印 pts_time
                  f"scale=640:-1"),
          "-fps_mode", "vfr", os.path.join(frames_dir, "raw_%05d.jpg"),
          "-hide_banner", "-loglevel", "error"])
    return len(glob.glob(os.path.join(frames_dir, "raw_*.jpg")))
```

`frames_meta.txt` 形如：
```
frame:0    pts:112640  pts_time:12.5
frame:1    pts:135168  pts_time:15.0
...
```
第 i 段（0-based）按顺序对应 `raw_00001.jpg`, `raw_00002.jpg`…（ffmpeg 输出序号 1-based，一一对齐）。

> **风险与兜底**（写进代码注释）：`metadata=print` 的段序与 `raw_%05d.jpg` 的写盘序在单趟 vfr 下是一致的；但若某版 ffmpeg 行为异常，`timestamps.py` 解析后会校验"段数 == raw 帧数"，不一致则退化为 `t = index * fps_floor` 的近似估算并在 `caveats` 标注（NFR-1 诚实）。

## 4. 意图分类（`intent.py`）—— FR-2 / G4

```python
def classify(prompt: str, has_image: bool) -> IntentResult:
    # FR-2.2: 有图强信号 → 默认 VISUAL_LOCATE, 但仍让 LLM 用 prompt 细化 return_scope
    # FR-2.4: 规则兜底优先跑, 命中直接返回, 省一次 LLM 调用
    rule = _rule_based(prompt, has_image)
    if rule is not None:
        return rule
    # LLM few-shot 分类 (classify 档 provider + 结构化输出)
    return llm.text_structured("classify", prompt, has_image, IntentResult)
```

- **规则兜底 `_rule_based`**（FR-2.4）：`has_image` → VISUAL_LOCATE；命中"什么时间/第几秒/最早/出现在" → LOCATE；"有没有/是否/检测/包含" → MODERATE；"总结/讲了什么/概括/主要内容" → SUMMARY。规则只处理高置信度单标签场景；模糊或多意图交给 LLM。
- **LLM 分类**：走 `classify` 档 provider（Claude Haiku / GPT-5.5-mini / GLM-flash 之一，config 可换）+ §6.4 结构化输出；system 里放四类定义 + 3-4 个 few-shot（你举的四个例子正好当 few-shot）。**纯文本任务，即使 provider 不支持 vision 也能承担此角色。**
- **VISUAL_LOCATE 的参考图文字描述**（FR-2.3）：由 `visual.py` 在分析阶段按需生成（一次 vision 档调用），不在分类阶段做，避免无谓开销。

## 5. 管线规划（`planner.py`）—— FR-3

纯函数、纯规则，把需求 §FR-3 的表直接编码：

```python
def plan(intents: list[Intent]) -> Plan:
    # 多意图取"最严格"并集: 只要含 LOCATE/MODERATE/VISUAL_LOCATE 就要时间戳+密采样
    dense = any(i in (Intent.LOCATE, Intent.MODERATE, Intent.VISUAL_LOCATE) for i in intents)
    moderate = Intent.MODERATE in intents
    if not dense:                                   # 纯 SUMMARY
        return Plan(intents, fps_floor=1.0, scene=0.30, dedup_threshold=8,
                    max_frames=150, need_timestamps=False, grid_label_time=False,
                    keep_audio=False)
    return Plan(intents,
                fps_floor=0.5, scene=0.20,
                dedup_threshold=0 if moderate else 3.5,   # MODERATE 关去重(§6.3)
                max_frames=None if moderate else 400,     # MODERATE 不封顶→分段
                need_timestamps=True, grid_label_time=True,
                keep_audio=moderate)                       # 暴力/尖叫声学线索
```

> 数值取自需求 §FR-3 的"初始建议"，集中在 `config.py` 便于实测调优。

## 6. 多模型 LLM 适配层（`llm/`）—— FR-5 / §6.5，原生 HTTP + 多 provider

**这是本次多模型改造的核心。** 不用官方 SDK，全部走原生 HTTP；上层 analyzer 只依赖统一门面，不知道背后是 Claude 还是 GPT/GLM。

### 6.1 统一门面与抽象基类（`llm/base.py`, `llm/__init__.py`）

上层只调三个函数，provider 内部差异全部被吸收：

```python
# ---- 统一的 provider 无关请求/响应类型 (llm/base.py) ----
@dataclass
class ImageRef:
    path: str                      # 本地帧路径, provider 内部各自 base64 编码
    cacheable: bool = False        # 参考图等固定前缀, 支持 caching 的 provider 会利用

@dataclass
class LLMResult:
    text: str                      # 原始文本 (结构化时是 JSON 串)
    usage: dict                    # {input_tokens, output_tokens, cache_read_tokens?} 尽力而为
    model: str
    raw: dict                      # provider 原始响应, 便于排查

class LLMProvider(ABC):
    """一个 provider = 一种线格式 + 一个鉴权 + 一组能力开关。"""
    name: str
    caps: "Capabilities"           # vision / json_schema / prompt_cache / batch 是否支持
    @abstractmethod
    def chat(self, *, system: str, messages: list[Msg],
             images: list[ImageRef] = (), max_tokens=4096,
             json_schema: dict | None = None) -> LLMResult: ...

# ---- 上层唯一入口 (llm/__init__.py) ----
def get_provider(role: str) -> LLMProvider:
    """role ∈ {'vision','classify'}; 从 config 的 provider 注册表解析出具体 provider+model。"""

def vision_structured(role, system, images, instruction,
                      schema: type[BaseModel], cacheable_images=()) -> BaseModel:
    """门面: 选 provider → 组装请求 → 结构化输出(见 6.4) → Pydantic 校验。analyzer 只调这个。"""
```

### 6.2 两个适配器（覆盖 Claude / GPT-5.5 / GLM-5.2）

业界视觉+对话 API 只有两种线格式，因此**两个适配器即全覆盖**：

**`AnthropicProvider`（`llm/anthropic.py`）** — `POST {base_url}/v1/messages`
```python

           "content-type": "application/json"}

      {"type":"image","source":{"type":"base64","media_type":"image/jpeg","data":b64}},
      {"type":"text","text":instruction}]}],
}
# 响应: resp["content"][i]["text"]; usage 在 resp["usage"]
```

**`OpenAICompatProvider`（`llm/openai_compat.py`）** — `POST {base_url}/v1/chat/completions`
覆盖 **GPT-5.5、GLM-5.2 及一切 OpenAI 兼容端点**（智谱、国产网关、vLLM 等）：
```python

  "model": model,
  "messages": [
    {"role":"system","content":system},

### 6.4 跨 provider 结构化输出（`llm/structured.py`）—— 替代 SDK 的 parse

不同 provider 结构化能力不一，用"能力优先 + 统一兜底"：

```python
def structured(provider, system, images, instruction, schema: type[BaseModel]):
    js = pydantic_to_json_schema(schema)
    if provider.caps.json_schema:
        # 走原生 json_schema / response_format, 拿到即校验
        r = provider.chat(system=system, images=images,
                          messages=[user(instruction)], json_schema=js)
    else:
        # 兜底: 把 schema 内联进 instruction, 要求"只输出 JSON"
        r = provider.chat(system=system, images=images,
                          messages=[user(instruction + "\n严格按此 JSON schema 输出, 只输出 JSON:\n" + dumps(js))])
    obj = _extract_json(r.text)            # 容忍 ```json fences / 前后噪声
    try:
        return schema.model_validate(obj)
    except ValidationError:
        # 失败重试 1 次, 把校验错误回灌给模型 (对任意 provider 都有效)
        r2 = provider.chat(system=system, images=images,
                           messages=[user(instruction), assistant(r.text),
                                     user("上次输出不符合 schema, 错误如下…请只输出修正后的 JSON")])
        return schema.model_validate(_extract_json(r2.text))
```

这样即使某个 provider 完全不支持结构化输出，也能拿到合法对象。

### 6.5 Prompt caching（能力可选，NFR-3）

- **仅当 `provider.caps.prompt_cache` 为真时启用**（目前 Anthropic 支持）。`ImageRef.cacheable=True` 的图片 + system 由 `AnthropicProvider` 打 `cache_control`。
- **VISUAL_LOCATE**：参考图不变 → 标 `cacheable=True`，每个候选帧只付变化部分（缓存读 ~0.1×）。**Claude 上省钱的关键。**
- 不支持 caching 的 provider（GPT/GLM 视官方而定）：自动跳过，功能不受影响，只是不省这笔。遵守前缀不变量：system 里不放时间戳/帧号。

### 6.6 批处理（能力可选，`llm/batch.py`）

- **仅当 `provider.caps.batch` 为真时启用**。MODERATE 逐帧高召回（§7.3）走批处理省钱：Anthropic Batches / OpenAI Batch，两种线格式各一实现，`custom_id=frame名`，轮询到完成收结果。
- 不支持 batch 的 provider（如某些 GLM 网关）：**自动降级为同步分段并发**（`concurrent.futures` 限流），贵一点但功能不变。

### 6.7 重试与降级（自研，因为没有 SDK 兜底）

- **HTTP 层**：`llm/base.py` 统一对 429/5xx 做指数退避重试（读 `Retry-After`），连接错误重试；`requests`/`httpx` 都设**总体超时**（避免逐块超时假死）。
- **拒答**：Anthropic 的 `stop_reason=="refusal"` / OpenAI 的 `finish_reason=="content_filter"` → 统一映射为"无法判定，建议人工复核"，不重试原 prompt（合规 §8）。
- **上云告知（C-4）**：`Orchestrator` 首次调用前打印一次"帧将发送到 <provider> 服务商"提示，并标明当前 provider 名。

### 7.1 SUMMARY（`summary.py`）—— FR-5.1

def analyze(out_dir, plan) -> Answer:

### 7.2 LOCATE / VISUAL_LOCATE 的两阶段骨架（`locate.py` + `rescan.py`）—— FR-5.2/5.4, §6.2

两者共用"粗扫定区间 → 精扫到秒"骨架，差异仅在"命中判定"（文字 vs 参考图）：

```python
def two_stage_locate(out_dir, plan, hit_fn, source_video) -> Answer:
    stamps = timestamps.load(out_dir)                    # G1
    # 阶段1 粗扫: 按时间序把带时间戳标签的拼图交给 hit_fn, 找目标"大致落点"
    coarse_hit = coarse_scan(out_dir, stamps, hit_fn)    # → 命中帧 or None
    if coarse_hit is None:
        return assembler.not_found(out_dir, plan.caveats())   # FR-6.2 诚实否定
    # 阶段2 精扫: 只在命中帧 ±window 秒区间, ffmpeg 密集重抽(每0.2-0.5s), 逐帧判定首次出现
    fine_frames = rescan.dense_extract(source_video, out_dir,
                                       t0=coarse_hit.t - W, t1=coarse_hit.t + W, step=0.3)
    first = earliest_hit(fine_frames, hit_fn)            # 精确到秒
    return assembler.from_locate(first, out_dir, plan.caveats())
```

- `hit_fn`（LOCATE）：`lambda frame: llm.vision_structured("vision", target_system, [frame], "是否出现<target>?", HitSchema)`
- `hit_fn`（VISUAL_LOCATE）：见 §7.4，先像素预筛再 vision 模型语义确认。
- `rescan.dense_extract`：对候选区间用 `ffmpeg -ss t0 -to t1 -vf fps=1/step` 重抽，并**同样落 pts_time**（复用 §3 机制），保证精扫帧也有时间戳。
- **R-2 分辨率**：精扫阶段可对候选帧用更高 `scale`（如 1280）或裁剪目标区域再识别（车牌类），在 `rescan` 里作为参数。

### 7.3 MODERATE（`moderate.py`）—— FR-5.3, §6.3, §8

```python
def analyze(out_dir, plan, target) -> Answer:
    stamps = timestamps.load(out_dir)                    # 关去重后帧很多
    # 分段(chunk)高召回: 每帧一个判定; provider 支持 batch 则走批处理(省), 否则同步并发(§6.6)
    verdicts = llm.judge_frames(                          # 内部自动 batch/并发降级
        role="vision",
        system=prompts.moderate_system(target),          # §8: 只判定"是否存在", 不描述细节
        frames_with_time=stamps, schema=ModerateSchema)
    hits = [v for v in verdicts if v.present and v.confidence >= THRESH]  # 高召回
    ranges = merge_adjacent(hits)                         # 命中片段合并为时间区间
    return assembler.from_moderate(ranges, out_dir, plan.caveats())
```

- 合规约束全部体现在 `prompts.moderate_system`：要求模型只回 `{present: bool, confidence: float, t}`，**不生成/不细节化**露骨内容（C-1）；涉未成年人命中即高优先级标注建议人工复核（C-2）。

### 7.4 VISUAL_LOCATE（`visual.py`）—— FR-5.4, §6.6 三级由粗到细

```python
def analyze(out_dir, plan, query_image, source_video) -> Answer:
    stamps = timestamps.load(out_dir)
    # 级1 预筛(本地零依赖): 复用 crv 16×16 RGB 像素相似度, 取 Top-K 候选
    scored = similarity.rank(query_image,
                             [out_dir/"frames"/s.frame for s in stamps])  # §6.6-1
    topk = scored[:K]                                     # K≈15, config 可调
    # (可选) 生成参考图文字描述, 交叉验证 (FR-2.3)
    desc = llm.describe("vision", query_image)            # 一次 vision 调用, 写入 target 回显
    # 级2 语义确认(多模态): 参考图+候选帧成对给 vision 模型(参考图设 cacheable), 判"同一对象?"
    def hit_fn(frame):
        return llm.vision_structured(
            role="vision",
            system=prompts.visual_system(desc),           # 含"区分同一 vs 相似"(§6.6歧义)
            images=[query_image, frame],                  # 参考图 cacheable=True (支持 caching 的 provider 自动利用)
            instruction="右图是否出现左图中的同一对象? 返回 same/similar/no + 相似度",
            schema=VisualHitSchema, cacheable_images=[query_image])
    # 级3 精扫: 复用 §7.2 骨架, 对最早确认命中的候选做区间密采精确到秒
    return two_stage_locate_over(topk, hit_fn, source_video, out_dir, stamps)
```

- `similarity.rank`（`similarity.py`）：直接搬 `core.py:120-128` 的 `sig`/`pct_diff`，把"差异%"取反做相似度排序。**零新依赖。**
- **R-5 相似≠同一**：`VisualHitSchema` 用三档 `same/similar/no`，assembler 据此**降级措辞并调低 confidence**（§6.6 歧义要求）。

## 9. 主编排（`orchestrator.py`）

```python
class Orchestrator:
    def ask(self, source, prompt, query_image=None, *, out_dir="mingcha-out",
            cookies=None, cookies_from_browser=None) -> Answer:
        # FR-1 输入校验(源可达/图片可读/受支持格式), 上云告知(C-4)
        _validate(source, query_image)
        # ① 分类 (FR-2)
        intent = intent.classify(prompt, has_image=query_image is not None)
        log_echo(intent)                                  # FR-2.5 回显可纠正
        # ② 规划 (FR-3)
        plan = planner.plan(intent.intents)
        # ③ 预处理: 复用 core.process, 传入 plan 的抽帧参数; 产出 frames+timestamps+grids
        result = core.process(source, out_dir,
            scene=plan.scene, fps_floor=plan.fps_floor,
            dedup_threshold=plan.dedup_threshold, max_frames=plan.max_frames or 100000,
            keep_audio=plan.keep_audio, cookies=cookies,
            cookies_from_browser=cookies_from_browser,
            emit_timestamps=plan.need_timestamps,         # 【新增开关】
            grid=True, grid_label_time=plan.grid_label_time)
        if query_image: shutil.copy(query_image, out_dir/"query_image.jpg")
        # ④ 分析: 按意图分发
        ans = _dispatch(intent, plan, out_dir, result, prompt, query_image)
        # ⑤ 组装 + 落盘 answer.json (FR-6)
        return assembler.write(ans, out_dir)
```

> `core.process` 需新增 `emit_timestamps` / `grid_label_time` 两个可选开关（默认 False，保证老 `crv` 命令零行为变化）。当 `emit_timestamps=True` 时，`process` 内部串起 §3 的 `parse_meta → dedup_frames(stamps=…) → timestamps.write`。

## 11. 成本与性能（NFR-3 / R-1）

| 意图 | 主要开销 | 控制手段 |
|---|---|---|
| SUMMARY | 1 次 vision（所有拼图） | 拼图已 ~1/9 图量；单次调用（provider 上下文小则内部分批） |
| LOCATE | 粗扫 N 张 + 精扫区间 M 张 | 两阶段（§6.2）；粗扫用拼图批量；精扫仅候选区间 |
| VISUAL_LOCATE | 预筛(本地免费) + Top-K 次 vision | 像素预筛把逐帧→Top-K；参考图 caching（支持的 provider） |
| MODERATE | 逐帧，帧最多 | 批处理省 50%（支持的 provider）/ 否则同步并发 + system 缓存 |

- **provider 无关的省钱手段**（任何模型都受益）：两阶段扫描、本地像素预筛、拼图降图量、分段并发。
- **provider 相关的省钱手段**（能力可选，自动降级）：prompt caching、批处理——不支持则跳过，功能不受影响。
- **护栏**（`config.py`）：`MAX_FRAMES_HARD`（如 2000，超出告警并 `log` 丢弃说明——NFR-1 不静默截断）、`TOPK`、精扫 `WINDOW`/`STEP`、并发数、单 provider 图片数/上下文上限。
- **预算可选**：`mingcha ask --budget-frames N` 上限，超出走更稀采样并在 `caveats` 声明。

## 12. 错误处理与日志

- 沿用 crv `_run` 静默执行 + 文件存在性兜底；但 `ask` 路径对关键步骤（下载失败/无帧/无时间戳）**主动抛出可读错误**（FR-1.5），不静默。

- LLM 拒答 / 超时 / 配额 → 分类：可重试（`llm/base.py` 自研退避）、需人工（MODERATE 敏感）、硬失败（明确报错）。
- 日志：每意图打印所选 `Plan` 参数 + **当前 provider/model**（NFR-5 可解释）、每次 LLM 调用的 `usage`（成本可见；支持 caching 的 provider 额外打印缓存命中 token）。

## 13. 测试策略

无既有测试框架（CLAUDE.md），沿用"真实视频手动验证"，但补三类可复现夹具：

## 15. 未决 & 需你拍板

1. **GLM-5.2 / GPT-5.5 官方参数**：设计已把它们做成配置驱动（§6.3 注册表），但**真实 base_url / model-id / 是否支持 vision·json_schema·batch 需你或官方文档确认**后填表。尤其：这两个型号是否都支持**多图/图文混合**输入？若某个不支持 vision，它只能做意图分类。
2. **CLIP 初筛器**：MVP 用像素相似度（零依赖）。是否在 P4 引入 CLIP（准确率↑，但引 torch 依赖 ~GB 级）？倾向"作为 `[clip]` 可选 extra"。
3. **OCR**：车牌类 LOCATE 是否值得接 OCR（如 PaddleOCR）做候选帧二次识别以提升车牌准确率？倾向 P2 之后按需。
4. **批处理时延**：MODERATE 用批处理省 50% 但可能等较久（各家不同）。若要求实时，改同步分段并发（贵但快）。需你定"审核场景能否接受批处理时延"——也可做成 `--no-batch` 开关。
5. **参考图多对象**：一张参考图含多个对象时如何指定"找哪个"——靠 `--image` + prompt 文字补充，还是加交互框选？MVP 先靠 prompt。
6. **默认 provider**：默认 `claude`（vision 成熟、支持 caching/batch），还是你更希望默认国产 `glm`（合规/成本）？改 `config.ROLES` 一行即可。
