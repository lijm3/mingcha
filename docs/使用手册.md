## 1. 它能做什么

给明察一个视频（本地文件或 URL）加一句话要求，它会自动判断你的意图，抽取关键帧做多模型分析，最后给出**带时间戳和截图证据**的答案。

| 意图 | 你会这样问 | 你会得到 |
|---|---|---|
| **SUMMARY** 理解 | "总结这个视频讲了什么" | 结构化摘要（主题 / 分段脉络 / 关键点） |
| **LOCATE** 定位 | "琼A 车牌最早出现在什么时间" | 精确到秒的时间戳 + 命中帧 |
| **MODERATE** 审核 | "有没有暴力画面" | 高召回判定 + 命中时间区间 |
| **VISUAL_LOCATE** 以图搜 | 附一张图 + "这个人出现在哪" | 时间戳 + 相似度 + 同一/相似判别 |

核心特点：全程携带时间戳，答案可回溯到源视频的具体时间点；多模型可插拔（Claude / GPT / GLM），原生 HTTP 无 SDK 绑定。

## 2. 安装

### 2.1 前置：系统依赖

明察依赖以下命令行工具，需自行安装并确保在 `PATH` 上（**不能用 pip 装**）：

| 工具 | 用途 | 是否必需 |
|---|---|---|
| `ffmpeg` / `ffprobe` | 抽帧、探测、转音频 | **必需** |
| `whisper` | 语音转写（无字幕时） | 可选 |

验证：

```bash
ffmpeg -version
ffprobe -version
```

> Windows 可用 `winget install ffmpeg` 或从 ffmpeg.org 下载后把 `bin/` 加进 PATH；macOS 用 `brew install ffmpeg`。

### 2.2 安装明察

```bash
# 基础安装（yt-dlp、Pillow 会自动装上）
pip install -e .

# 需要语音转写时（视频无内嵌/外挂字幕）
pip install -e ".[whisper]"
```

安装后会得到两个等价命令：`mingcha` 和简写 `mc`。

验证：

```bash
mingcha --version
```

## 3. 快速开始

```bash
# 最简单的一条：总结一个本地视频
mc ask video.mp4 "总结这个视频讲了什么"

# 总结一个网络视频（URL 走 yt-dlp 下载）
mc ask "https://example.com/watch?v=xxxx" "讲了什么"
```

运行时你会看到：上云告知 → 识别出的意图 → 预处理统计（抽了多少帧、去重后多少、拼了几张图、转写情况）→ 每一步的 token 用量 → 最终答案。

答案同时会落盘到 `mingcha-out/answer.json`。

## 4. 四类意图详解

明察会**自动分类**你的意图（先走关键词规则，命中不了再让模型判断）。你也可以通过措辞主动引导。

### 4.1 SUMMARY（理解 / 总结）

触发词示例：总结、讲了什么、概括、主要内容、介绍一下

```bash
mc ask lecture.mp4 "这个讲座主要讲了哪些点"
```

产出：主题一句话 + 按时间顺序的分段脉络 + 关键结论。若视频有字幕/转写，会一并作为依据。

### 4.2 LOCATE（定位某目标最早出现的时间）

触发词示例：什么时间、第几秒、最早、出现在、何时、几分几秒、时间点、定位

```bash
mc ask dashcam.mp4 "红色的车最早出现在什么时间"
```

工作方式：**两阶段**。先按关键帧粗扫找到大致命中，再在该点 ±8 秒内密集重抽（更高分辨率，利于车牌/文字二次识别），把时间精确到秒级。找不到会**如实说找不到**，并声明采样局限。

### 4.3 MODERATE（内容审核）

触发词示例：有没有、是否、检测、包含、有无、存在吗、审核

```bash
mc ask clip.mp4 "有没有暴力或血腥画面"
```

工作方式：**高召回**优先（阈值偏低，宁可多报也少漏），逐帧判定后把相邻命中点合并成时间区间。会保留音轨供有听觉能力的模型参考（尖叫等声学线索）。

> ⚠️ 审核结果是自动初筛，**以人工复核为准**。涉及疑似未成年人或模型拒答的帧会被标注为"高优先级人工复核"。

### 4.4 VISUAL_LOCATE（以图搜视频）

用 `--image` 附一张参考图即可触发（有图就是强信号）。

```bash
mc ask surveillance.mp4 "这个人出现在视频的哪些时刻" --image suspect.jpg
```

工作方式：**三级由粗到细**。① 本地零依赖的像素预筛取 Top-K 候选（省多模态调用）→ ② 参考图与候选帧成对送模型做语义确认 → ③ 对最早命中做精扫。

**重要**：明察会严格区分"**同一个体**"和"**外观相似的同类**"。无独有特征佐证时只会判"相似"，并相应降低措辞与置信度——相似不等于同一。

## 5. 命令行参考

```
mc ask <source> [prompt] [选项]
```

| 参数 / 选项 | 说明 | 默认 |
|---|---|---|
| `source` | 视频 URL 或本地路径（必填） | — |
| `prompt` | 一句话要求 | 空（此时必须给 `--image`） |
| `--image PATH` | 参考图，触发 VISUAL_LOCATE | 无 |
| `-o, --out DIR` | 输出目录（覆盖式） | `mingcha-out` |
| `--provider NAME` | 一次性把所有角色切到某 provider（如 `glm`） | 无 |
| `--vision-model SPEC` | 仅画面分析用某模型，形如 `openai:gpt-5.5` | 无 |
| `--classify-model SPEC` | 仅意图分类用某模型 | 无 |
| `--cookies FILE` | Netscape 格式 cookies 文件 | 无 |
| `--cookies-from-browser B` | 从浏览器读 cookies（如 `chrome`） | 无 |
| `--no-cache` | 强制重算，忽略缓存复用 | 关 |
| `--version` | 打印版本 | — |

`prompt` 和 `--image` 至少要给一个，否则退出。

## 6. 模型与 Provider 配置

明察内置三个 provider：

| provider | 类型 | 用途 | 密钥来源 |
|---|---|---|---|
| `claude` | Anthropic | 默认画面分析 + 分类 | 已内置（第三方代理） |
| `openai` | OpenAI 兼容 | GPT 系列 | 已内置（第三方代理） |
| `glm` | OpenAI 兼容 | 智谱 GLM | 环境变量 `ZHIPU_API_KEY` |

### 6.1 角色 → 模型

明察内部有两个"角色"：`vision`（画面分析，默认 `claude:claude-opus-4-8`）和 `classify`（意图分类，默认 `claude:claude-haiku-4-5`）。

### 6.2 三种切换方式（优先级从高到低）

```bash
# ① 只切某个角色（最细）
mc ask v.mp4 "定位车牌" --vision-model openai:gpt-5.5
mc ask v.mp4 "总结" --classify-model glm

# ② 一次性把所有角色切到某 provider
mc ask v.mp4 "总结" --provider glm

# ③ 用配置文件覆盖（见下）
```

`SPEC` 写法：`provider:model`（如 `openai:gpt-5.5`）或只写 `provider`（用该 provider 的默认模型）。

### 6.3 用配置文件覆盖（无需改代码）

在**运行目录**下建 `.crv/providers.yaml` 或 `.crv/providers.json`，可覆盖 provider 的 `base_url`/`model`/能力开关，以及角色映射：

```json
{
  "providers": {
    "glm": { "model": "glm-4v-plus" }
  },
  "roles": {
    "vision": "glm:glm-4v-plus"
  }
}
```

> YAML 需要额外装 `pip install pyyaml`；不想装就用 `.json`（零依赖）。密钥**永远不从这里读**，只走环境变量，避免密钥进配置文件。

### 6.4 关于密钥

- `glm` 需要你自己设 `export ZHIPU_API_KEY=...`。
- `claude` / `openai` 的密钥已内置在 `config.py` 中（走第三方代理 camel-hub / denxio）。
- ⚠️ **安全提醒**：内置密钥会随源码存在，请勿把带真实密钥的 `config.py` 提交到公开仓库或分享出去。

## 7. 输出产物说明

所有产物写入 `-o` 指定的目录（默认 `mingcha-out/`，**每次运行覆盖**）：

| 文件 / 目录 | 内容 |
|---|---|
| `answer.json` | **最终答案**（结构化，见下） |
| `source.mp4` | 下载或复制来的源视频 |
| `frames/frame_NNN.jpg` | 去重后的关键帧（按时间顺序编号） |
| `grids/grid_NN.jpg` | 3×3 关键帧拼图，格子标注了时间戳 |
| `timestamps.json` | 帧→秒 映射（`{frame, t, hms}`） |
| `transcript.txt` | 语音转写（有字幕或跑了 whisper 时） |
| `rescan/` | LOCATE/VISUAL 精扫的临时高清帧 |
| `audio.m4a` | 完整音轨（MODERATE 时保留） |
| `query_image.jpg` | 归档的参考图（VISUAL_LOCATE 时） |
| `.mingcha_cache.json` | 缓存指纹（内部用） |

### `answer.json` 字段

```json
{
  "intent": "LOCATE",
  "target": "红色的车",
  "query_image": null,
  "answer": "目标「红色的车」最早出现在 00:00:12.500（约第 12.5 秒）。",
  "evidence": [
    { "frame": "frame_007.jpg", "t": 12.5, "hms": "00:00:12.500",
      "confidence": 0.9, "similarity": null, "note": "画面右侧" }
  ],
  "confidence": 0.9,
  "caveats": "定位基于关键帧采样……更短暂的出现可能未被抽到。",
  "artifacts_dir": "/abs/path/mingcha-out"
}
```

- `evidence` 每条都能对回具体帧和时间；`similarity` 仅 VISUAL_LOCATE 有值。
- `caveats`（局限声明）在否定结论时**必然非空**——这是明察"诚实优先"的设计：找不到就说找不到，并说明采样限制。

## 8. Python API

```python
from mingcha import ask

# 等价于 Orchestrator().ask(...)
answer = ask("video.mp4", "总结这个视频讲了什么")
print(answer.answer)
for e in answer.evidence:
    print(e.hms, e.note)

# 以图搜 + 自定目录 + 关缓存
answer = ask("v.mp4", "这个人出现在哪",
             query_image="ref.jpg", out_dir="run1", use_cache=False)
```

需要更多控制时直接用 `Orchestrator`：

```python
from mingcha.orchestrator import Orchestrator

orch = Orchestrator(vision_model="glm")           # 只把画面分析切到 GLM
ans = orch.ask("v.mp4", "有没有敏感内容", out_dir="out")
```

返回的是一个 `Answer`（Pydantic 模型），字段同 `answer.json`。

## 9. 缓存机制

同一个 `(视频源 + 问题 + 管线 + 模型)` 组合第二次运行时，若上次产物完好，明察会**直接复用结果**，跳过抽帧和模型调用（省时省 token）。

- 本地文件的指纹含 mtime 和大小，文件改了会自动失效。
- 想强制重算：加 `--no-cache`（CLI）或 `use_cache=False`（API）。
- 换了模型或换了问题都会重新计算。

## 10. 处理需要登录的视频

对于需要账号才能访问的**你本人有权访问**的内容，可以带 cookies：

```bash
# 方式一：从浏览器直接读
mc ask "https://..." "总结" --cookies-from-browser chrome

# 方式二：用导出的 Netscape cookies 文件
mc ask "https://..." "总结" --cookies cookies.txt
```

下载流程会三级回退：先无 cookies 试 → 再浏览器 cookies → 再 cookies 文件。

> 请仅对自己有权访问的内容使用 cookies。

## 11. 常见问题与排错

**Q: 报错 "未找到 yt-dlp" / "视频源不存在"**
A: URL 下载需要 `yt-dlp`（`pip install yt-dlp`）；本地路径要确认文件真实存在。

**Q: 下载失败（私有视频？）**
A: 目标可能需要登录，见[第 10 节](#10-处理需要登录的视频)带 cookies。

**Q: 转写显示 "install whisper to transcribe"**
A: 视频没有可用字幕，需要语音转写请装 `pip install -e ".[whisper]"` 并确保 `whisper` 在 PATH。视频本身无音轨则无法转写。

**Q: 某个 provider 报 "缺少 API key"**
A: 对 `glm` 需设 `ZHIPU_API_KEY` 环境变量；自定义 provider 请检查 `.crv/providers` 配置。

**Q: 时间戳提示"退化估算"**
A: ffmpeg 元数据的帧数与实际抽帧数对不上，明察退回用 `序号×步长` 近似，时间点可能有偏差——这会写进 `caveats` 如实告知。

**Q: LOCATE/VISUAL 没找到，但我确定画面里有**
A: 关键帧采样有密度限制，一闪而过的目标可能没被抽到。可换更精确的措辞，或针对性截取该片段再问。

**Q: 想验证流程但不想烧 token**
A: 跑测试即可：`pytest`（全部 mock 模型，不联网、不需密钥）。

## 12. 使用限制与免责

- **审核（MODERATE）仅为高召回自动初筛**，可能误报，最终判定需人工复核；涉未成年人或敏感内容尤其如此。
- **以图搜（VISUAL_LOCATE）区分"同一"与"相似"**，判"相似"时不代表就是同一对象，请结合证据自行判断。
- 分析会把关键帧**上传到所选模型服务商**（运行时有明确告知）。请勿用于处理你无权分析的隐私内容。
- 结果的时间精度受抽帧密度限制，`caveats` 字段会如实声明每次的采样局限。
