# 明察（MingCha）前后端分离 —— 详细设计文档

> 版本 v1.1 ｜ 2026-07-09
> 目标形态：**本地运行**（不依赖 Docker）的前后端分离 Web 应用，让不会用命令行的普通人也能通过友好界面使用明察。
> 关联文档：`docs/使用手册.md`（CLI 用法）、根目录 `CLAUDE.md`（现有架构）、`docs/实现进度.md`。
>
> **v1.1 修订（前端落地缺陷修复）**：① SSE 业务错误改名 `fail`，与原生 `EventSource` 的 `error` 事件解耦（§5.4 / §8.6）；② 大文件上传改用 `XMLHttpRequest` 带进度（§8.8）；③ `AnswerOut` 补 `summary_detail` / `query_image_url`、`EvidenceOut` 补 `verdict`，让 `SummaryCard` / 参考图回显 / VISUAL 徽章有数据来源（§5.3 / §9.7）；④ 前端类型改由 `openapi-typescript` 自动生成，杜绝前后端契约漂移（§8.9）。

---

## 0. 一句话概述

把现有的 `Orchestrator.ask()` CLI 能力，包一层 **FastAPI 异步任务服务**（后端）+ **React 单页应用**（前端）。装好依赖后用**标准命令启动后端**（uvicorn），前端构建一次由后端一并托管，浏览器打开 `http://localhost:8000` 即可上传视频/粘贴链接、选意图、看进度、看带截图证据的结果。**不依赖 Docker、不依赖 nginx**。模型 API Key 由**用户在界面自己填**，只存在浏览器本地与请求内存中，**不落盘、不进日志**。

## 1. 目标与非目标

### 1.1 目标
- **本地易启动**：装好依赖后，`uvicorn` 一条命令起后端（前端预构建后由后端托管）；ffmpeg 可用 `pip install static-ffmpeg` 免手装，不需要 Docker/nginx。
- **普通人可用**：图形界面完成"选视频 → 提问 → 看结果"，无需终端、无需懂 provider/模型。
- **前后端分离**：后端提供 REST + SSE，前端纯静态 SPA，二者可独立开发/替换。
- **复用现有内核**：不重写分析逻辑，最小侵入地把 `Orchestrator` 从"同步 + print"改造为"异步 + 进度回调 + 请求级 Key 注入"，**保持 CLI 仍可用**。
- **用户自带 Key**：界面填 Key，服务端不持有默认 Key，不承担 token 成本。
- **四类意图全支持**：SUMMARY / LOCATE / MODERATE / VISUAL_LOCATE（含参考图上传）。
- **证据可视化**：结果里的时间戳/截图可点击，视频播放器跳转到对应秒。

### 1.2 非目标（本期明确不做）
- ❌ 账号 / 注册 / 登录 / 多租户隔离（无账号，单机自用）。
- ❌ 公网多用户防滥用体系（限流、配额、计费、SSRF 强隔离）——本地默认只监听回环地址，风险面小。
- ❌ 对象存储 / 分布式队列 / 横向扩缩容（本地单机，文件存本地卷，内存队列足够）。
- ❌ 服务端持有默认 Key（成本全在用户侧）。

> 📌 **未来若要公网化**：见 [§11.3 公网化 Checklist](#113-未来公网化需要补齐的清单)，本设计已预留边界，不会推倒重来。

## 2. 术语

| 术语 | 含义 |
|---|---|
| **任务 (Task)** | 一次完整的"视频 + 提问 → 答案"分析，有唯一 `task_id` 和独立工作目录 |
| **意图 (Intent)** | SUMMARY / LOCATE / MODERATE / VISUAL_LOCATE 四类 |
| **产物 (Artifact)** | 一次任务产生的文件：帧、拼图、截图、answer.json、timestamps.json 等 |
| **证据 (Evidence)** | 答案中可回溯的 `{frame, t, hms, confidence, similarity, note}` |
| **进度事件 (Progress Event)** | 经 SSE 推给前端的阶段/百分比/日志 |
| **Provider Key** | 用户自带的模型密钥（claude / openai / glm 各一个，可只填用到的） |

### 3.4 技术选型

| 层 | 选型 | 理由 | 被否决的备选 |
|---|---|---|---|
| 后端框架 | **FastAPI + uvicorn** | 原生 async、SSE 友好、Pydantic 与现有 `types.py` 同源、自动 OpenAPI 文档 | Flask（异步/SSE 支持弱）；Django（过重） |
| 任务执行 | **ProcessPoolExecutor + asyncio.Queue** | 分析是 CPU 密集（ffmpeg/whisper）+ 阻塞，须进程隔离；本地单机无需 Redis | Celery+Redis（本地部署过重，还得额外装 Redis）；线程池（GIL + 无法真正取消 subprocess） |
| 进度推送 | **SSE (Server-Sent Events)** | 单向服务端→客户端，浏览器原生 `EventSource`，比 WebSocket 简单，天然适配进度流 | WebSocket（双向能力用不上，握手/重连更复杂）；轮询（延迟高、浪费） |
| 前端框架 | **React 18 + Vite + TS** | 用户指定；生态最大，视频播放器/上传/时间轴组件成熟 | — |
| 前端状态 | **Zustand** | 轻量、样板少、异步友好，适合"任务 + 进度 + 结果"这类少量全局状态 | Redux（样板过多）；仅 Context（跨组件进度更新易踩性能坑） |
| 存储 | **本地文件系统卷** | 本地部署，产物即文件；`data/tasks/{id}/` 天然隔离 | S3/MinIO（本地过重） |
| 打包 | **后端 uvicorn 一体托管前端静态 + API + SSE**（单进程） | 无需 Docker/nginx；ffmpeg 用 pip 的 `static-ffmpeg` 免手装；前端预构建 `dist` 由后端托管，同源无跨域 | docker compose（要装 Docker，普通人门槛高——仅作可选见 §10.3）；前后端各起一进程（多一个服务，日常使用没必要） |

## 4. 部署形态：本地运行（不依赖 Docker，不做一键脚本）

最朴素的方式：装好前置依赖后，用标准命令**分别启动后端与前端**；上线时前端构建一次、由后端一并托管即可。**不需要 Docker、不需要 nginx、不需要一键脚本**——就是常规的 Python 服务 + 前端两步。

### 4.1 前置依赖

| 依赖 | 用途 | 安装方式 |
|---|---|---|
| **Python 3.10+** | 后端服务 | python.org |
| **Node 18+** | 前端构建/开发 | nodejs.org（**仅需要改前端时**；若用发布包内置的 `frontend/dist`，可跳过） |
| **ffmpeg / ffprobe** | 抽帧/探测 | 见下方三选一 |

**ffmpeg 是普通人最容易卡住的一步**，给三种途径，推荐第 ②：
1. 系统安装：`winget install ffmpeg`（Win）/ `brew install ffmpeg`（Mac）/ `apt install ffmpeg`（Linux），确保在 PATH。
2. **pip 安装（推荐，免手装）**：`pip install static-ffmpeg`，它自带 ffmpeg + ffprobe 二进制；后端启动时调用 `static_ffmpeg.add_paths()` 注入 PATH，`media.py` **零改动**即可用（见 §4.4）。
3. 手动下载二进制，把所在目录加入 PATH。

### 4.2 启动后端

```bash
cd mingcha
python -m venv .venv && . .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install ./src ./backend "yt-dlp>=2024.1.0" static-ffmpeg
# 可选转写: pip install openai-whisper
uvicorn app.main:app --app-dir backend --port 8000
# 后端默认 --host 127.0.0.1, 只本机可达
```

环境变量可选（见 §13.1），不设也能跑。

### 4.3 启动前端（两种）

**A. 开发模式**（改前端时用，热更新）：
```bash
cd frontend && npm install && npm run dev        # http://localhost:5173
# vite.config.ts 里把 /api 与 /artifacts 代理到 http://localhost:8000
```

**B. 生产模式**（日常使用，推荐）：构建一次，交给后端托管，只需开一个后端进程：
```bash
cd frontend && npm install && npm run build       # 产出 frontend/dist
# 之后只跑后端即可，浏览器开 http://localhost:8000
```
> 发布给不懂前端的人时，把预构建好的 `frontend/dist` 一并打进发布包，对方连 Node 都不用装，只需 §4.2 起后端。

### 4.4 后端一体托管前端 + ffmpeg 注入（`backend/app/main.py`）

生产模式下后端同时提供 API、SSE、静态产物**和前端页面**，单进程、同源、无跨域、无需 nginx：

```python
from fastapi import FastAPI
from fastapi.staticfiles import StaticFiles

# ① 启动时把 static-ffmpeg 的二进制注入 PATH, media.py 无需改动
try:
    import static_ffmpeg
    static_ffmpeg.add_paths()
except ImportError:
    pass                                    # 未装则依赖系统 PATH 里的 ffmpeg

app = FastAPI()
app.include_router(meta_router)             # /api/health ...
app.include_router(tasks_router)            # /api/tasks ...
app.mount("/artifacts", ArtifactStatic(), name="artifacts")   # 产物(带 token 校验)
# SPA 兜底放最后: 其余路径交给前端 index.html
app.mount("/", StaticFiles(directory="frontend/dist", html=True), name="spa")
```

- **SSE 无需特殊配置**：uvicorn 对流式响应默认不缓冲，进度事件实时下发（这正是省掉 nginx 的前提）。
- 上传大小限制在后端应用层校验（§6.6），不依赖反代。

### 4.5 网络与安全默认
- uvicorn 默认 `--host 127.0.0.1`，**只有本机能访问**，符合本地自用定位。
- 想让局域网其它设备访问：显式 `--host 0.0.0.0`（文档需提示：此时你填的 Key 与产物会对同网段可达，谨慎）。

## 5. 后端 API 设计

所有业务端点前缀 `/api`。Key 通过 **请求头** 传，绝不进 URL/query。

### 5.1 端点总表

| 方法 | 路径 | 说明 | 关键请求 | 响应 |
|---|---|---|---|---|
| GET | `/api/health` | 健康检查 + 能力探测（ffmpeg/whisper 是否可用） | — | `200` `{status, ffmpeg, whisper, version}` |
| POST | `/api/tasks` | 创建分析任务（上传或 URL） | multipart 或 json + `X-Provider-Keys` | `202` `{task_id, task_token}` |
| GET | `/api/tasks/{id}` | 查询任务状态 | `?token=` | `200` `TaskStatus` |
| GET | `/api/tasks/{id}/events` | **SSE** 进度流 | `?token=` | `text/event-stream` |
| GET | `/api/tasks/{id}/answer` | 获取最终结果 | `?token=` | `200` `Answer` / `409` 未完成 |
| GET | `/artifacts/{id}/{path}` | 静态产物（帧/拼图/视频/参考图） | `?token=` | 文件流 |
| POST | `/api/tasks/{id}/cancel` | 取消任务 | `?token=` | `200` `{cancelled}` |
| POST | `/api/keys/verify` | （可选）校验某 Key 是否可用 | `X-Provider-Keys` | `200` `{claude:true, ...}` |

### 5.2 创建任务请求（两种源）

**A. 本地上传**（`multipart/form-data`）：
```
POST /api/tasks
Headers: X-Provider-Keys: {"claude":"sk-...","openai":"sk-..."}
Fields:
  video: <file>                 # 视频文件
  prompt: "总结这个视频"          # 可空(纯以图搜时)
  image: <file>                 # 可选, VISUAL_LOCATE 参考图
  intent: "auto"                # auto|SUMMARY|LOCATE|MODERATE|VISUAL_LOCATE
  provider: ""                  # 可选, 一次性切 provider(如 glm)
  vision_model: ""              # 可选, 如 openai:gpt-5.5
```

**B. URL 源**（`application/json`）：
```json
POST /api/tasks
{ "url": "https://...", "prompt": "讲了什么", "intent": "auto",
  "provider": null, "vision_model": null,
  "cookies_from_browser": null }
```

### 5.3 Pydantic 模型（`backend/app/schemas.py`）

```python
class ProviderOverride(BaseModel):
    provider: str | None = None          # 一次性切所有角色
    vision_model: str | None = None      # 形如 "openai:gpt-5.5"
    classify_model: str | None = None

class CreateTaskJson(BaseModel):
    url: str
    prompt: str = ""
    intent: Literal["auto","SUMMARY","LOCATE","MODERATE","VISUAL_LOCATE"] = "auto"
    override: ProviderOverride = ProviderOverride()
    cookies_from_browser: str | None = None

class CreateTaskResponse(BaseModel):
    task_id: str
    task_token: str                      # 访问该任务产物的凭证

class TaskStatus(BaseModel):
    task_id: str
    state: Literal["queued","downloading","extracting","transcribing",
                   "analyzing","assembling","done","error","cancelled"]
    progress: float                      # 0~1
    stage_note: str = ""                 # 人类可读, 如 "去重后 40 帧"
    intent: str | None = None            # 分类后回填
    error: str | None = None
    created_at: float
    caveats: str = ""

# 复用现有 types.py 的 Answer/Evidence, API 层再包一层产物 URL
class EvidenceOut(BaseModel):
    frame: str
    frame_url: str                       # /artifacts/{id}/frames/frame_007.jpg?token=
    t: float
    hms: str
    confidence: float
    similarity: float | None = None      # 仅 VISUAL_LOCATE
    verdict: str | None = None           # 仅 VISUAL_LOCATE: "same"|"similar", 供徽章区分同一个体/同类外观
    note: str = ""

class SummaryDetail(BaseModel):          # 仅 SUMMARY: 结构化摘要, 供 <SummaryCard> 三段式渲染
    topic: str = ""                      # 主题 / 一句话概括
    segments: list[str] = []             # 分段脉络(按时间顺序)
    key_points: list[str] = []           # 关键要点

class AnswerOut(BaseModel):
    intent: str
    target: str | None
    answer: str                          # 完整摘要/结论文本(始终有)
    summary_detail: SummaryDetail | None = None   # 仅 SUMMARY: 三段式结构化(数据来源见 §9.7)
    evidence: list[EvidenceOut]
    confidence: float
    caveats: str
    video_url: str                       # 供播放器跳转
    query_image_url: str | None = None   # 仅 VISUAL_LOCATE: 参考图回显(来自 Answer.query_image)
    grids: list[str] = []                # 拼图 URL
```

> **字段来源约定**：`*_url` 由 API 层用 `task_token` 拼装，**不落进 `answer.json`**。`summary_detail`（SUMMARY 三段式）与 `verdict`（VISUAL same/similar）目前内核 `Answer`/`Evidence` **尚未产出**，需按 §9.7 补齐透传；在补齐前前端对 `null` **优雅降级**（`SummaryCard` 退化为整段 `answer`，`verdict` 徽章隐藏）。

### 5.4 SSE 进度协议

端点 `GET /api/tasks/{id}/events?token=` 返回 `text/event-stream`，事件类型：

```
event: state
data: {"state":"extracting","progress":0.4,"stage_note":"去重后 40 帧(自 64)"}

event: log
data: {"line":"转写: whisper 完成"}

event: done
data: {"answer_url":"/api/tasks/abc/answer?token=..."}

event: fail
data: {"error":"下载失败(私有视频? 需 cookies)","state":"error"}
```

> ⚠️ **业务失败事件叫 `fail`，绝不叫 `error`**：原生 `EventSource` 把 `error` 保留给**连接层故障**（断线、网络抖动、服务重启），该事件 `event.data` 为 `undefined` 且会触发浏览器**自动重连**。若服务端也发 `event: error`，前端就无法区分"任务真失败"与"连接暂断"，会把本可自愈的抖动误判为失败并中断进度流。故业务终止统一走 `fail`。

前端 `EventSource` 监听 `state` / `log` / `done` / `fail`；收到 `done` / `fail` 即主动 `close()`。**原生 `error`（连接层）不 `close`**，交给浏览器按 `Last-Event-ID` 自动重连，UI 仅提示"连接中断，重连中…"；服务端保留最近事件即可续传（本地场景可简化为重发当前快照）。前端实现见 §8.6。

### 5.5 错误映射

| 内核情况 | HTTP | 说明 |
|---|---|---|
| 源不存在 / 图不可读（`_validate` 抛错） | `400` | 请求即时校验，不建任务 |
| 下载失败 / ffmpeg 缺失 | 任务内 `error` 事件 | 已建任务，异步阶段失败 |
| 找不到目标（`not_found`） | `200` + `caveats` | **诚实否定不是错误**，正常返回 |
| Key 缺失 / 无效（`LLMError`） | 任务内 `error` | 提示"请检查对应 provider 的 Key" |
| 模型拒答（`LLMRefusal`） | 单帧降级，任务正常完成 | 结果标注"建议人工复核" |

## 6. 异步任务与进度

### 6.1 为什么必须异步

一次分析是**分钟级**（下载 + 抽帧 + whisper 转写 + 多次 LLM 调用），且大量 ffmpeg/whisper 是**阻塞 + CPU 密集**。若在请求线程内同步跑，会占满 uvicorn worker、HTTP 超时、无法反馈进度。故：请求只**创建任务并立即返回** `task_id`，实际执行放进程池，进度经 SSE 推送。

### 6.2 任务执行模型

```
FastAPI(async) ──创建──▶ TaskManager
                          ├─ tasks: dict[task_id, TaskRecord]   # 内存任务表
                          ├─ ProcessPoolExecutor(max=CONCURRENCY)
                          └─ 每任务一个 asyncio.Queue(进度事件)

进程池 worker(独立进程) 执行 service.run_task(...)
   └─ 通过 multiprocessing 管道把进度事件回传主进程 → 塞进该任务的 Queue
SSE 端点 async 循环 await queue.get() → yield 给浏览器
```

选 `ProcessPoolExecutor` 而非线程：ffmpeg/whisper 是子进程 + CPU 密集，进程隔离才能真正并发、真正**取消**（终止子进程），也避免单任务崩溃拖垮服务。

### 6.3 任务状态机

```
queued ──▶ downloading ──▶ extracting ──▶ transcribing ──▶ analyzing ──▶ assembling ──▶ done
   │            │              │               │               │             │
   └────────────┴──────────────┴───────────────┴───────────────┴─────────────┴──▶ error
   └──(用户取消, 任意阶段)──────────────────────────────────────────────────────▶ cancelled
```

进度百分比映射（经验值，可调）：queued 0 → downloading 0.1 → extracting 0.4 → transcribing 0.55 → analyzing 0.75 → assembling 0.95 → done 1.0。

### 6.4 进度回调注入（内核改造见 §9）

现状 `orchestrator.py` / `preprocess.py` 用 `print(...)` 输出进度。改造为**可选 `progress` 回调**：

```python
# service.py 里定义回调, 把内核进度转成 SSE 事件
def make_progress(emit):
    def cb(state: str, fraction: float, note: str = ""):
        emit({"state": state, "progress": fraction, "stage_note": note})
    return cb

orchestrator.ask(source, prompt, ..., progress=cb)
```

内核里把原来的 `print(f"预处理: {n} 帧...")` 改为 `progress and progress("extracting", 0.4, f"{n} 帧")`，**同时保留 print 供 CLI**（`progress` 为 None 时行为不变）。

### 6.5 取消

`POST /api/tasks/{id}/cancel` → TaskManager 设置取消标志 + 终止该任务的进程池 future / 子进程。内核在关键循环点（逐帧 judge、抽帧前）检查 `cancel_event`，抛 `TaskCancelled`。ffmpeg/whisper 子进程通过记录 PID 后 `terminate()` 结束。

### 6.6 资源护栏（本地精简）

| 护栏 | 默认 | 位置 |
|---|---|---|
| 最大并发任务 | 2 | `MINGCHA_MAX_CONCURRENCY`，进程池大小 |
| 单任务超时 | 30 min | TaskManager 计时，超时转 error + 清子进程 |
| 上传大小 | 500 MB | 后端应用层校验（读流时累计字节，超限即中断） |
| 帧数硬上限 | 2000 | 复用现有 `config.MAX_FRAMES_HARD` |
| 产物 TTL | 24 h | 后台定时清理 `data/tasks/` 过期目录 |

### 6.7 任务与产物 TTL 清理

后端启动一个 `asyncio` 定时任务（或 APScheduler），每小时扫描 `data/tasks/`，删除 `created_at` 超过 `TASK_TTL_HOURS` 的目录及内存任务记录。本地磁盘不至于无限增长。

## 7. 用户自带 Key（本地简化版）

### 7.1 Key 生命周期与信任边界

```
┌── 浏览器(前端) ──────────────┐        ┌── 后端(内存) ────────────────┐
│ Key 设置面板输入             │        │ 请求处理期间: dict 驻留        │
│ 存 localStorage(可选"记住")  │ HTTPS/ │ 注入 Orchestrator/get_provider│
│ 每次请求放 X-Provider-Keys ──┼──本地──▶│ 任务结束 → del, 不落盘        │
│ 明文仅在本机内存/存储        │  回环  │ 绝不写日志/answer.json/磁盘    │
└─────────────────────────────┘        └──────────────────────────────┘
                                        信任边界: 本机进程内
```

原则（**必须**）：
- Key 只经 **请求头** `X-Provider-Keys`（JSON：`{"claude":"...","openai":"...","glm":"..."}`），**不进 URL、不进 query、不进日志**。
- 后端**不持久化** Key：不写数据库、不写文件、不写进 `answer.json` 或任务记录；仅在处理该任务的进程内内存驻留，任务结束即释放。
- 日志脱敏：任何异常/访问日志遇到 Key 字段一律 `sk-****`（`keys.py` 提供 `redact()`）。
- 前端"记住 Key"仅存浏览器 `localStorage`（用户本机），并明确提示"仅保存在本机，清除浏览器数据即失效"。

### 7.2 本地便利：环境变量预置（可选）

因为是本地自用，允许**后端 `.env` 预置 Key**（如 `ANTHROPIC_API_KEY`），此时界面可不填。优先级：**请求头用户 Key > 后端环境变量**。这样"自己用"填一次环境变量，"给别人用"则各自界面填。

### 7.3 请求级注入（替代硬编码）

现状 `config.py` 把 Key 硬编码进 `PROVIDERS`。改造为**请求级注入**（详见 §9）：API 层把 `X-Provider-Keys` 解析成 `runtime_keys` 字典，穿过 `Orchestrator` → `get_provider` → provider 实例，覆盖 config 默认。**移除源码中的明文 Key**。

### 8.6 SSE 封装

```ts
// 关键: 原生 'error' = 连接层故障(无 data), 交给浏览器自动重连, 不 close;
//       业务失败走服务端自定义的 'fail' 事件(见 §5.4), 收到才 close。
function useSSE(taskId: string, token: string, onEvent: (e: SSEEvent) => void) {
  useEffect(() => {
    if (!taskId || !token) return
    const es = new EventSource(`/api/tasks/${taskId}/events?token=${token}`)
    es.addEventListener('state', e => onEvent({ type: 'state', ...JSON.parse(e.data) }))
    es.addEventListener('log',   e => onEvent({ type: 'log',   ...JSON.parse(e.data) }))
    es.addEventListener('done',  e => { onEvent({ type: 'done', ...JSON.parse(e.data) }); es.close() })
    es.addEventListener('fail',  e => { onEvent({ type: 'fail', ...JSON.parse(e.data) }); es.close() })
    // 原生连接错误: e.data 为 undefined。不 close —— EventSource 会带 Last-Event-ID 自动重连,
    // 仅把 UI 切到"连接中断, 重连中…"。只有服务端 'fail' 或 'done' 才真正结束流。
    es.onerror = () => onEvent({ type: 'disconnected' })
    return () => es.close()
  }, [taskId, token])
}
```

> `SSEEvent` 的 `type` 需含 `'fail'` 与 `'disconnected'`：`ProgressView` 收到 `disconnected` 显示"重连中…"而非报错，仅 `fail` 才落错误态。这正是 §5.4 把业务错误从 `error` 改名为 `fail` 的落地原因——两类事件在前端被彻底分开处理。

### 8.7 i18n & 空/错态
- 中文优先（文案与内核 caveats 一致）。
- 空态：未选源时按钮禁用 + 引导；URL 下载失败提示 cookies 选项；无 whisper 时提示"该视频无字幕，装 whisper 或换有字幕的视频"。

### 8.8 大文件上传（带进度）

500 MB 视频上传是**分钟级**，而 SSE 进度要等任务创建成功后才开始。若用 `fetch` 提交 multipart，它对**上传进度**支持很差（无 `upload.onprogress`），用户点"开始"后会对着无反馈的界面干等——误以为卡死。故**创建任务这一步改用 `XMLHttpRequest`**（axios 亦可，底层同为 XHR）读取上传进度：

```ts
function createTaskWithProgress(
  form: FormData, keys: object, onUp: (f: number) => void,
): Promise<CreateTaskResponse> {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest()
    xhr.open('POST', '/api/tasks')
    xhr.setRequestHeader('X-Provider-Keys', JSON.stringify(keys))          // Key 仍走请求头, 不进 URL
    xhr.upload.onprogress = e => { if (e.lengthComputable) onUp(e.loaded / e.total) }  // 上传进度 0~1
    xhr.onload  = () => xhr.status === 202
      ? resolve(JSON.parse(xhr.responseText))
      : reject(new Error(`create task failed: ${xhr.status}`))
    xhr.onerror = () => reject(new Error('network error'))
    xhr.send(form)
  })
}
```

**两段式进度**（前端单调递增，不回退）：
1. **上传段**（`uploadProgress` 0→1）：仅此段用 XHR 的 `upload.onprogress`，界面显示"上传中 xx%"。
2. 上传完成拿到 `task_id` → 立即 `useSSE` 订阅 → 切到**分析段**（`status.progress`，映射见 §6.3）。
3. URL 源（无文件上传）跳过上传段，直接进分析段。

> ⚠️ 别用 `fetch` 传大文件 multipart：拿不到上传进度，在"普通人可用"定位下是体验硬伤。

### 9.7 结果 schema 扩展（支撑前端 SummaryCard / VISUAL 回显）

```python
    orch = Orchestrator(
        provider=payload["override"].get("provider"),
        vision_model=payload["override"].get("vision_model"),
        runtime_keys=payload["keys"],          # 用户自带 Key
    )
    try:
        ans = orch.ask(payload["source"], payload["prompt"],
                       query_image=payload.get("image"),
                       out_dir=payload["out_dir"],
                       cookies_from_browser=payload.get("cookies_from_browser"),
                       use_cache=False, progress=progress,
                       cancel=lambda: cancel_flag["v"])
        emit({"type":"done"})
        return {"ok": True}
    except Exception as e:
        emit({"type":"error", "error": f"{type(e).__name__}: {e}"})
        return {"ok": False}
```

### 10.2 产物静态服务

### 10.3 可选：Docker 容器化（进阶/想隔离环境的人）
本地运行**不需要** Docker（见 §4）。但若你更习惯容器、或想把环境完全隔离，可自行加一个 `Dockerfile`：`python:3.12-slim` + `apt install ffmpeg` + `pip install ./src ./backend` + 复制 `frontend/dist`，`CMD uvicorn app.main:app`。单镜像即含前后端（后端托管 dist），`docker run -p 8000:8000 -v ./data:/data` 起。属于可选便利，非默认路径。

## 11. 安全与隐私

### 11.1 本地部署必做项
1. **去硬编码 Key**（§9.2）——现存最大隐患，务必先做。
2. **Key 不落盘、不进日志**（§7.1），日志脱敏。
3. **默认绑回环** `127.0.0.1`（§4.1），不主动暴露公网。
4. **产物访问需 task_token**（§10.2）+ 路径穿越防护。
5. **上传校验**：大小上限、扩展名/魔数校验（是视频）、防止写出目录外。
6. **TTL 清理**（§6.7）：视频与产物默认 24h 删除。
7. **上云告知**：界面明示"关键帧将上传到你填的模型服务商"；MODERATE 结果附"高召回初筛，以人工复核为准"。

### 11.2 本地场景可接受的简化
- 无账号、无鉴权（除 task_token 防误访问）——因默认仅本机可达。
- SSRF：本地自用风险低，仅做基础超时 + 明显内网地址提示（不做强隔离）。

### 11.3 未来公网化需要补齐的清单
> 本设计的边界（task_token、请求级 Key、独立目录、TTL）已为此预留。若要上公网，需补：
- HTTPS/TLS 强制、绑 `0.0.0.0` + 反代限流（IP 滑窗、并发配额）。
- **SSRF 强防护**：URL 下载白/黑名单、禁内网段、DNS 重绑定防护。
- 上传更严格的沙箱与配额；任务级速率限制。
- Key 传输的额外加固（虽仍不落库，但考虑前端加密/一次性令牌）。
- 结构化审计日志（脱敏）、异常上报、资源监控。
- 可选：轻量账号 + 配额（若要控成本/防滥用）。

### 13.2 关键依赖
- 后端：`fastapi`、`uvicorn[standard]`、`python-multipart`（上传）、`sse-starlette`（可选，简化 SSE）、`static-ffmpeg`（免手装 ffmpeg）、`yt-dlp`、`mingcha`（本地 path 安装）。可选：`openai-whisper`（转写）。
- 前端：`react`、`react-dom`、`zustand`、`react-dropzone`、`react-router-dom`（可选）；开发依赖 `openapi-typescript`（从 `/openapi.json` 生成 TS 类型，见 §8.9）；构建产物 `frontend/dist` 由后端托管。
- 系统：`ffmpeg`/`ffprobe`（推荐用 `static-ffmpeg` pip 包，或系统安装）。**无需 Docker、nginx、Node（用预构建 dist 时）**。
