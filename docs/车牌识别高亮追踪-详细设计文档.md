# 明察（MingCha）· 车牌识别与高亮追踪 详细设计文档

> 本文是「视频中车辆车牌识别 + 高亮标注跟随移动」这一新功能的落地设计。它在 [视频智能体-详细设计文档.md](./视频智能体-详细设计文档.md) 定义的四类意图（SUMMARY / LOCATE / MODERATE / VISUAL_LOCATE）之上，新增**第五类意图 `PLATE`**，并为明察引入第一条「空间维度 + 时序追踪 + 视频回写」的分析链路。

> **阅读前提**：熟悉明察的五层编排（分类 → 规划 → 预处理 → 分析 → 组装）、G1 时间戳地基、诚实优先（NFR-1）、永不崩、config 驱动等既有约定。本文所有小节都标注了对应的 FR/G/R 编号，沿用主设计文档的编号体系。
>
> **本文只谈设计，不含代码改动。** 实现路线见 §15。

## 0. 关键技术决策（已与需求方确认，先定）

主设计文档 §0 把 OCR 定位为「LOCATE 车牌/文字类的**可选二次识别**插件（P2 之后）」、把这条记为优化项 **R-2**（`config.py:64` 的 `RESCAN_SCALE_WIDTH=1280` 注释即为其预埋伏笔）。本功能把 R-2 **升格为一等意图 `PLATE`**，并明确了它与纯 VLM 范式的分野。以下决策已确认：

| 决策点 | 结论 | 理由 |
|---|---|---|
| **车牌场景** | **中国车牌为主**（蓝牌 / 黄牌 / 绿色新能源 / 双层黄牌），预留通用车牌扩展位 | 用国产专用模型（HyperLPR3）准确率远高于通用 OCR；通用/国外车牌作为 R 级扩展。 |
| **技术范式** | **专用 CV 为主力 + VLM 兜底纠错（混合）**，非纯 VLM | VLM 给不出精确像素框、逐帧密集调用又慢又贵、更做不了帧间追踪；「高亮跟随移动」必须靠 CV 检测 + 多目标追踪。VLM 退居「模糊车牌兜底纠错」，发挥明察既有多模型优势。 |
| **依赖策略** | CV 依赖（`opencv` / `onnxruntime-gpu` / `hyperlpr3` 等）全部进**可选 extra `[plate]`**；核心包保持「完全自包含、mock 测试不烧 token」 | 延续 `[whisper]` / `[clip]` 的隔离手法；未装 extra 时走可读降级提示，不破坏现有 9 项测试与 CLI 行为。 |
| **运行环境** | 默认按 **GPU 可用**优化（`onnxruntime-gpu`），同时保留 CPU 保底路径 | 逐帧密集检测在 GPU 上快一个量级；CPU 环境自动降级为「降采样帧率 + 轻量模型」并在 caveats 声明。 |
| **交付目标** | **一步到位产出标注视频** `annotated.mp4`（检测 + 追踪 + OCR + 多帧融合 + 高亮回写），而非只出静态证据 | 需求方明确要「高亮显示并跟随车辆移动」的最终形态。 |
| **对付车牌模糊** | 组合拳：**① 多帧时序投票（主）→ ② 车牌语法规则约束 → ③ 低置信触发超分 → ④ 图像预处理 → ⑤ VLM 兜底纠错** | 见 §6.5–§6.9。多帧投票是追踪白送的红利、性价比最高；VLM 兜底是明察差异化。 |
| **诚实边界** | 模糊到多帧投票 + VLM 都拿不准的车牌，给低 confidence + caveats 声明「车牌模糊无法确认」，**绝不硬猜**（NFR-1） | 与现有 `assembler.not_found` 同款诚实语义。 |

## 1. 需求边界

### 1.1 做什么（FR-5.5 PLATE）

- **FR-5.5.1 车牌检测**：在视频每个（密采）关键帧上定位所有车牌的像素框（bbox）。
- **FR-5.5.2 车牌识别（OCR）**：读出每个车牌的字符串 + 车牌颜色（蓝/黄/绿）+ 逐字符置信度。
- **FR-5.5.3 跨帧追踪**：把同一辆车/同一车牌在连续帧间关联为一条轨迹（`track_id`），即使中途短暂遮挡也尽量续上。
- **FR-5.5.4 多帧融合定案**：对每条轨迹的多帧 OCR 结果做字符级投票，定一个最终车牌号 + 置信度（对付模糊，§6.5）。
- **FR-5.5.5 高亮回写**：把每辆车的 bbox + 最终车牌号画回视频，标注**跟随车辆移动**，输出 `annotated.mp4`（保留原音轨）。
- **FR-5.5.6 证据与诚实**：每条车牌轨迹作为一条空间证据（bbox + track_id + plate_text + 出现区间）落入 `answer.json`；识别不出如实否定并声明采样/清晰度局限。

### 1.2 不做什么（划清边界，避免范围蔓延）

- 不做**车型/品牌/颜色车身**识别（只做车牌）。
- 不做**跨视频的车牌库比对 / 布控告警**（无数据库、无检索）。
- MVP 不做**实时流**处理（只处理离线文件/URL，与明察现状一致）。
- 不做**车牌打码/脱敏**输出（与「高亮显示」诉求相反；如需可作 R 级反向选项）。
- 不承诺模糊车牌 100% 可读——走诚实优先，给置信度与局限。

### 1.3 隐私与合规声明（NFR）

车牌属敏感信息。文档层面明确：本功能仅用于用户**本人授权**的视频分析（与 `media.fetch_video` 的 cookies「仅限本人授权内容」声明一致）；输出的 `answer.json` 与 `annotated.mp4` 含明文车牌，落盘目录沿用 `.gitignore` 排除策略（`*-out/`），不入版本库。

## 2. 与现有架构的关系：为什么这是一条新链路

明察现有范式是「**时间维度的稀疏采样 + VLM 语义分析**」：抽帧 → 去重抽稀（≤150 帧）→ 拼图 → 一次/逐帧 VLM 调用 → 结构化文本。证据 `Evidence` 只有 `frame / t / hms / confidence`——**是时间坐标，没有空间坐标**。

车牌高亮追踪需要三个现有链路都不具备的能力：

| 能力 | 现状 | 缺口 | 本文对策 |
|---|---|---|---|
| 车牌**空间定位** | Evidence 无 bbox | 需要像素框 | `types.py` 扩 `BBox` + `Evidence.bbox`（§4） |
| **跟随移动**（同一车跨帧） | `dedup_timed` 去重把连续帧抽稀丢了 | 需要逐帧密集 + 追踪 track_id | planner 新增 `PLATE` 专用密采 plan（§5.2）+ ByteTrack（§6.3） |
| **高亮回写视频** | 只落 `answer.json` + 截图 | 需要把标注画回每帧再编码 | 新增 `annotate.py`（§7） |

> **命名：G6 空间-时序地基。** 主设计文档把「每帧可靠对回源视频绝对时间」记为 G1 时间戳地基。本功能新增的「同一目标跨帧的空间框可靠关联」记为 **G6 空间-时序追踪地基**，与 G1 正交：G1 管「哪一秒」，G6 管「画面里哪个框、是不是同一个」。二者在 `PlateDetection` 里合流（既有 `t` 又有 `bbox` 又有 `track_id`）。

## 3. 系统分层与新增包结构

沿用主设计文档 §1 的五层编排，`PLATE` 意图接入方式与其它四类分析器**同构**：`orchestrator._dispatch` 按意图分发到 `analyzer/plate.py`，后者消费预处理产物 + 调 CV 模块 + 调 `assembler` 组装 `Answer`。

**设计原则（对齐明察工程审美）**：

1. **CV 依赖只在 `plate/` 子包里 `import`，且惰性**（函数内 import，与 `preprocess.py` 里 `from PIL import Image` 同款手法）。核心包与现有测试**零新增依赖**。
2. `analyzer/plate.py` 只依赖 `plate/` 门面 + `llm` 门面 + `assembler`，**不知道**背后是 HyperLPR3 还是 YOLO——与四个现有分析器只依赖 `llm` 门面同构。换检测/OCR 引擎 = 改 `plate/` 内部，不动 analyzer。
3. **G1 穿透**：`plate.py` 的密采帧全部携带绝对时间戳（复用 `rescan.dense_extract` 的 `-ss` 回加 t0 手法，§6.1），回写时用绝对时间对齐。

## 4. 数据结构（`types.py` 扩展）

在不破坏现有 `Evidence` / `Answer` 字段的前提下**增量扩展**（新字段全部可选、有默认值，旧分析器与旧 `answer.json` 不受影响）。

```python
class Intent(str, Enum):
    SUMMARY = "SUMMARY"
    LOCATE = "LOCATE"
    MODERATE = "MODERATE"
    VISUAL_LOCATE = "VISUAL_LOCATE"
    PLATE = "PLATE"                       # 【新】车牌识别 + 高亮追踪

# ---- G6 空间坐标 ----
class BBox(BaseModel):
    x: int; y: int; w: int; h: int        # 左上角 + 宽高，像素（源视频原分辨率坐标系）

# ---- Evidence 增量扩展（新字段可选，旧分析器不填）----
class Evidence(BaseModel):
    frame: str
    t: float
    hms: str
    confidence: float
    similarity: float | None = None       # 仅 VISUAL_LOCATE
    verdict: str | None = None            # 仅 VISUAL_LOCATE
    note: str = ""
    # —— 新增（仅 PLATE）——
    bbox: BBox | None = None              # 车牌在该帧的位置（代表帧）
    track_id: int | None = None           # 跨帧同一车牌标识
    plate_text: str | None = None         # 最终车牌号
    plate_color: str | None = None        # 蓝 / 黄 / 绿(新能源) / 双层黄

# ---- PLATE 逐帧检测（一辆车一帧一条）----
class PlateDetection(BaseModel):
    t: float                              # G1 绝对时间
    frame: str                            # 来源帧文件名
    bbox: BBox                            # G6 空间框
    text: str                             # 该帧 OCR 读数（未融合）
    char_confidences: list[float] = []    # 逐字符置信度（多帧投票依据）
    ocr_confidence: float = 0.0
    plate_color: str | None = None

# ---- PLATE 一条车牌轨迹（多帧融合后定案）----
class PlateTrack(BaseModel):
    track_id: int
    plate_text: str                       # 多帧投票 + 规则约束后的最终车牌号
    confidence: float                     # 融合置信度
    plate_color: str | None = None
    first_t: float                        # 首次出现（G1）
    last_t: float                         # 末次出现
    n_frames: int                         # 该轨迹的检测帧数（投票样本量）
    method: str = "vote"                  # vote | rule_fixed | superres | vlm_corrected
    detections: list[PlateDetection] = [] # 该轨迹全部检测（回写与复核依据）
    caveats: str = ""                     # 该车牌的个体局限（如「全程模糊」）

# ---- VLM 兜底纠错的结构化输出（复用 vision_structured 校验）----
class PlateVLMSchema(BaseModel):
    plate_text: str = ""
    plate_color: str | None = None
    confidence: float = 0.0
    reasoning: str = ""                   # 可解释：为何这么读（规则依据）

# ---- Answer 增量扩展 ----
class Answer(BaseModel):
    ...                                   # 现有字段不变
    annotated_video: str | None = None    # 【新】标注视频落盘路径（仅 PLATE）
    plate_tracks: list[PlateTrack] = []   # 【新】结构化车牌轨迹（仅 PLATE）
```

> **为何 `Evidence` 里放「代表帧」而 `PlateTrack` 里放「全部帧」**：`Evidence` 面向前端摘要展示（一条轨迹给一张最清晰帧 + 最终车牌号）；`PlateTrack.detections` 面向回写与人工复核（需要逐帧 bbox 做插值与追溯）。二者职责分离。

## 5. 意图接入（分类 / 规划 / 分发）

### 5.1 意图分类（`intent.py`，FR-2）

沿用「规则兜底优先，省一次 LLM」：

```python
_PLATE_KW = ("车牌", "车牌号", "牌照", "车号", "识别车牌", "高亮车牌", "标注车牌")
# _rule_based 中，PLATE 关键词优先级置于 LOCATE/MODERATE 之前
if any(k in p for k in _PLATE_KW):
    return IntentResult(intents=[Intent.PLATE], target=p,
                        return_scope="all_ranges", reason="命中车牌关键词")
```

LLM few-shot 分类的 system prompt（`prompts.CLASSIFY_SYSTEM`）补一条 PLATE 的示例；LLM 失败仍兜底 SUMMARY（永不崩，不变）。

### 5.2 管线规划（`planner.py`，FR-3）

`PLATE` 的采样诉求与现有意图**根本不同**：追踪要**连续帧**，不能去重抽稀。新增专用分支（不并入现有 dense 逻辑）：

```python
if Intent.PLATE in intents:
    return Plan(intents=intents,
                fps_floor=???,           # 见下：按 PLATE_SAMPLE_FPS，GPU 下更密
                scene=0.0,               # 关场景切割：车牌追踪要均匀连续，不要跳变触发
                dedup_threshold=0,       # 关去重（连续帧是追踪前提）
                max_frames=None,         # 不封顶（长视频分段处理，见 §6.1 护栏）
                need_timestamps=True,    # G1
                grid_label_time=False,   # 不拼 grids（PLATE 不走拼图 VLM 范式）
                keep_audio=True)         # 回写要保留原音轨
```

**关键差异**：`grid_label_time=False` 且不产出 grids——`PLATE` 不走「拼图喂 VLM」范式，`preprocess.run(..., want_grids=False)`（orchestrator 按意图传参）。这样预处理只产出**带时间戳的连续帧序列**，正是检测/追踪的输入。

**采样密度权衡**（`config.PLATE_SAMPLE_FPS`）：
- 太稀（如 1fps）→ 追踪断裂、快速驶过的车漏检、回写标注跳变。
- 太密（如原生 30fps）→ 帧数爆炸、GPU 也吃力、存储高。
- 折中：**检测/OCR 阶段用中等密度（如 5fps）**跑追踪与识别；**回写阶段另论**（§7 用 bbox 插值补齐到原始帧率，不必逐帧检测）。

### 5.3 分发（`orchestrator.py._dispatch`）

```python
elif it == Intent.PLATE:
    answers.append(plate.analyze(pre.out_dir, pl, target, pre.video, vision_model=vm))
```

多意图合并（`_merge_answers`）需处理 PLATE 的新字段：`annotated_video` / `plate_tracks` 取首个非空，`evidence` 并集（已有逻辑天然覆盖）。预处理传参需按意图关 grids（见 §5.2）——orchestrator 依 `Intent.PLATE in pl.intents` 传 `want_grids=False`。

## 6. 车牌分析器 `analyzer/plate.py`（主编排）

整体管线（对应 FR-5.5.1–5.5.4）：

```
密集抽帧(带时间戳,连续)              §6.1  ← 复用 preprocess/rescan，G1 穿透
  → 逐帧车牌检测 bbox               §6.2  ← plate.detect（HyperLPR3/YOLO）
  → 多目标追踪 track_id            §6.3  ← plate.track（ByteTrack）
  → 逐帧 OCR（每个 track 收多帧读数） §6.4  ← plate.ocr
  → 多帧字符级投票 → 每 track 定案   §6.5  ← plate.fuse（对付模糊·主力）
  → 车牌语法规则约束纠错            §6.6  ← plate.fuse（规则层）
  → 低置信 track 触发超分 + 复识别   §6.7  ← plate.superres（按需）
  → 仍低置信 → VLM 兜底纠错         §6.8  ← plate.vlm_fallback（明察差异化）
  → 组装 PlateTrack + 回写视频      §7    ← annotate + assembler.from_plate
```

**永不崩铺底**：任一 track 的检测/OCR/超分/VLM 失败，降级为「该 track 低置信 + caveats」，不中断其它 track，不中断整体（与 `visual._pair_judge` 的 `except → 降级 no` 同款）。

### 6.1 密集抽帧（连续帧，G1 穿透）

复用现有两个手法，不新造轮子：

- **抽帧命令**：沿用 `preprocess.extract_frames_timed` 的 `cwd=frames_dir + 相对文件名` 绕开 Windows filtergraph 盘符冒号转义（主设计文档「关键约定」）。但**去掉 `select='gt(scene,...)'`**（PLATE 要连续、不要场景跳选），改为纯 `fps=PLATE_SAMPLE_FPS` 均匀抽帧 + `metadata=print` 落 `pts_time`。
- **长视频分段**：`max_frames=None` 但受 `MAX_FRAMES_HARD` 护栏。超长视频**分段密采**（每段独立抽帧→检测→追踪，段间用车牌文本 + 末帧 bbox 做轨迹缝合）。分段是为控内存/存储，不是为省 token（PLATE 不烧 VLM token，除非走 §6.8 兜底）。
- **抽帧分辨率**：车牌是小目标，抽帧宽度需高于默认 640。取 `PLATE_FRAME_WIDTH`（默认 1280，对齐 `RESCAN_SCALE_WIDTH` 的「利于车牌识别」预埋）；bbox 坐标记录在**该抽帧分辨率坐标系**，回写时按缩放比还原到原始分辨率（§7）。

> **G1 铁律**：若采用 `-ss` 分段定位，`pts_time` 是相对段起点的，**必须回加段起点 t0 还原绝对时间**（与 `rescan.dense_extract` 完全一致的坑，主设计文档已警示）。这是 PLATE 时间正确性的命门。

### 6.2 车牌检测（`plate/detect.py`）

- **主引擎：HyperLPR3**。国产、专为中国车牌（蓝/黄/绿新能源/双层黄），检测 + 识别一体，`onnxruntime` 后端，GPU 下快、CPU 可跑。输出：每帧车牌 bbox + 四角点（用于 §6.9 透视矫正）+ 初步文本 + 颜色。
- **备选/增强：YOLO 车牌检测**（如在 CCPD 上训练的 YOLOv8-plate）+ 独立识别头。当 HyperLPR3 召回不足（小/远车牌）时可切换，属 R 级。
- **接口**（门面，稳定；内部引擎可换）：

```python
def detect(frame_path: str) -> list[PlateBox]:
    """一帧 → 若干车牌框（bbox + 四角点 + 颜色 + 初检文本 + 检测置信度）。
    引擎失败/未装 [plate] → 抛可读错误，由 analyzer 兜底为该帧空结果。"""
```

### 6.3 多目标追踪（`plate/track.py`）

- **算法：ByteTrack**（对遮挡/低分框鲁棒，无需额外 ReID 模型，轻量）。以「车牌 bbox 序列」为观测，跨帧输出稳定 `track_id`。
- **为什么追踪是模糊识别的前提**：一条 track 把「同一车牌的几十帧读数」聚到一起，才能做 §6.5 的多帧投票。没有 track，多帧信息无从关联。
- **缝合**：分段抽帧（§6.1）跨段时，用「上一段末帧 bbox + 文本」与「下一段首帧」做 IoU + 文本相似度缝合，避免同一辆车被拆成两条 track。
- **接口**：

```python
def track(dets_by_frame: list[tuple[float, str, list[PlateBox]]]) -> list[PlateTrack]:
    """按时间序的逐帧检测 → 分组为若干 track（此时 plate_text 尚未融合定案）。"""
```

### 6.4 逐帧 OCR（`plate/ocr.py`）

- 对每条 track 的每个检测框做 OCR（HyperLPR3 识别头），产出 `PlateDetection`：文本 + **逐字符置信度** + 颜色。逐字符置信度是 §6.5 投票的关键输入（很多 OCR 只给整体分，HyperLPR3 可取 per-char logits）。
- 已由检测阶段（§6.2 一体模型）顺带产出文本时，本步做**二次精识别**：对 §6.9 预处理/§6.7 超分后的车牌小图重新 OCR，通常比全帧检测顺带的读数更准。

### 6.5 多帧时序融合投票（`plate/fuse.py`）——对付模糊的主力

**核心思想**：车在动，同一车牌经过几十帧，每帧模糊的位置不同、总有相对清晰的时刻。把一条 track 的所有帧读数按**字符位对齐**，逐位做**置信度加权投票**，得到比任何单帧都准的最终车牌号。这是追踪白送的红利，几乎零额外成本。

```
输入：track 的 N 个 PlateDetection（每个含 text + char_confidences）
① 按车牌长度众数对齐（中国车牌普通 7 位 / 新能源 8 位）；长度异常帧降权或剔除
② 对每个字符位 i：
     candidates = { char: Σ(该帧在位 i 读到 char 的置信度) }
     final[i] = argmax(candidates)
     pos_conf[i] = max 得票 / 总票（该位可信度）
③ plate_text = join(final)；track.confidence = min(pos_conf) 或加权平均
④ 记录 method="vote"，n_frames=N（样本量越大越可信 → 反映到 confidence）
```

**边界处理**：
- N 很小（如 1–2 帧，车一闪而过）→ 投票无力，标记低置信 + caveats「样本不足」，交 §6.7/§6.8。
- 各帧长度不一致 → 先按长度众数归组，避免「7 位 vs 8 位」错位投票。
- 某字符位候选票数接近（如 `0` vs `D` 势均力敌）→ 标记为「不确定位」，优先送 §6.6 规则层 / §6.8 VLM 消歧。

### 6.6 车牌语法规则约束（`plate/fuse.py` 规则层）——零成本必做

中国车牌结构强、字符集受限，用规则纠正 OCR 高频混淆，几乎零成本救回大量错误：

- **结构约束**：`省份汉字 + 地区字母(A-Z) + 5位[字母数字]`（普通）；新能源 8 位且首位规则不同。位置决定字符集：第 1 位必是省份简称汉字集，第 2 位必是 A–Z。
- **混淆纠正**：结合位置约束消解 `0/O/D`、`1/I`、`8/B`、`6/G`、`5/S`、`2/Z`——例如「字母位读出数字 0」按上下文纠为 `O`/`D`，「数字位读出 O」纠为 `0`。
- **省份合法性**：首汉字校验在省份简称集合内；不在则标记可疑，送 VLM。
- 规则纠正后 `method="rule_fixed"`，并在 `reasoning` 记录改了哪几位（可解释，对齐 FR-2.5 精神）。

### 6.7 超分辨率（`plate/superres.py`）——低置信按需触发

- 仅对 §6.5 投票后仍低置信（`< PLATE_SUPERRES_THRESH`）的 track 触发，**不全量跑**（超分慢）。
- 引擎：**Real-ESRGAN**（或车牌专用超分）。对该 track 最清晰的 Top-K 帧的车牌小图放大重建 → 回 §6.4 重新 OCR → 回 §6.5 再投票。
- GPU 环境下代价可接受；CPU 环境默认关闭（`config` 开关）并在 caveats 声明。
- 成功提升则 `method="superres"`。

### 6.8 VLM 兜底纠错（`plate/vlm_fallback.py`）——明察差异化

超分后仍低置信、或存在「不确定位」的 track，走 VLM 兜底——这是明察相对纯 CV 方案的独有优势：

- **输入给 VLM**：该 track 最清晰的 2–3 张车牌裁剪图（或超分后的）+ 候选字符串 + 「这是中国车牌，请按 `省+字母+5位` 规则给最可能读数，逐位说明依据」的 instruction。
- **复用现有门面**：`llm.vision_structured("vision", plate_system(...), images=[...], schema=PlateVLMSchema, override=vision_model, cacheable_images=[...])`——与 `visual._pair_judge` 完全同款调用方式，支持 prompt caching 省钱。
- **拒答/异常**：`except → 保留投票结果 + caveats`（永不崩，同 `visual`）。
- **采纳策略**：VLM 结果与投票结果一致 → 提置信度；不一致 → 取 VLM（有规则推理）但在 caveats 标注「VLM 纠正」，`method="vlm_corrected"`。**这是唯一会烧 token 的环节**，仅对少数疑难 track 触发，成本可控。

### 6.9 图像预处理（`plate/preprocess_img.py`）——OCR 前处理

接在 §6.4 OCR 之前，便宜且有效：

- **透视矫正**：用 §6.2 的车牌四角点做透视变换，把拍歪的车牌摆正为矩形（对斜视角车牌提升显著）。
- **去运动模糊 + 锐化 + 对比度增强**：轻量图像增强；运动模糊可选 Wiener 反卷积。
- 全部作用于**车牌裁剪小图**，不动整帧，成本低。

## 7. 高亮回写 `annotate.py`（FR-5.5.5）——跟随移动的关键

产出 `annotated.mp4`：每辆车一个高亮框 + 车牌号标签，**框随车移动、标签用该 track 的最终定案文本**（不用每帧抖动的原始 OCR，避免标签闪烁）。

**流程**：

```
① ffmpeg 抽原始全帧序列（或按回写目标帧率，带 pts→帧号映射）
② 对每条 track：检测帧是稀疏的（§5.2 5fps），需在检测帧之间对 bbox 做
     线性插值（interpolate），补齐到回写帧率 → 框平滑跟随，不跳变
③ 逐帧绘制：
     - 用 track 的最终 bbox（插值后）画高亮框（颜色可按车牌色：蓝/黄/绿）
     - 标签固定为 track.plate_text（+ 可选置信度徽章）
     - 中文/省份汉字标签必须用 PIL ImageDraw 绘制（cv2.putText 不支持中文）
       —— 明察本就用 Pillow，直接复用
④ ffmpeg 把标注帧序列 + 原音轨（keep_audio）编码回 annotated.mp4
```

**关键坑与对策**：

- **坐标系还原**：检测在 `PLATE_FRAME_WIDTH`（1280）抽帧坐标系，回写在原始分辨率——bbox 按缩放比 `原宽/1280` 还原，否则框会错位。
- **中文标签**：`cv2.putText` 画不了汉字，用 `PIL.ImageDraw.text` + 字体文件（随包附带或探测系统字体），与 `preprocess.make_grids_timed` 画时间戳标签同款手法。
- **插值边界**：track 首/末检测帧之外不外推（车牌未出现/已离开），只在检测区间内插值+绘制。
- **性能护栏**：逐帧回写长视频代价高。`config` 给 `PLATE_ANNOTATE_MAX_SECONDS` 上限；超长视频只回写「有车牌出现的时间段」或降回写帧率，并在 caveats 声明。
- **无音轨**：`has_audio` 为假时跳过音轨合并（复用 `media.has_audio`）。

## 11. 诚实优先与永不崩（NFR-1 / 稳定性）在 PLATE 的落地

对齐主设计文档的两条铁律，逐点映射：

| 场景 | 行为 |
|---|---|
| 未检出任何车牌 | `assembler.from_plate([], …)` 诚实否定 + caveats「采样密度下未检出，短暂/远处车牌可能漏检」 |
| 检出但全部过糊 | 否定但**如实说明「检出 N 处车牌区域，均过于模糊无法可靠识别」**，附最清晰截图，不硬猜车牌号 |
| 单个 track OCR/超分/VLM 失败 | 该 track 降级低置信 + caveats，不影响其它 track，不中断整体 |
| 某帧检测崩（坏帧/解码错） | 跳过该帧，追踪照常（丢一帧不断链） |
| 未装 `[plate]` extra | 可读提示 Answer，不 crash |
| CPU 环境 / 无 GPU | 自动降级 + caveats 声明局限 |
| 回写超时/超长 | 只标注命中段或降帧率 + caveats，不无限跑 |
| 置信度语义 | `PlateTrack.confidence` 必须反映真实可读性；样本不足/模糊必须压低，不虚高 |

## 13. 性能、算力与成本预算

### 13.1 GPU 用途分布（吃不吃 GPU）

GPU **只用于「逐帧密集的神经网络推理」**——检测、OCR、（按需）超分三块；追踪 / 投票 / 规则是 CPU 算法，VLM 兜底走**云端 API 不占本地 GPU**。

| 环节 | GPU | 用来做什么 / 为什么 |
|---|---|---|
| 车牌检测（§6.2） | ✅ **主力** | 每帧跑一次检测网络找车牌框；5fps 密采下几分钟视频就上千帧，是全链路**最重的计算**，GPU 加速最明显 |
| 车牌 OCR（§6.4） | ✅ **主力** | 每个车牌小图跑识别网络读字符，逐帧密集 |
| 超分（§6.7） | ✅ **最吃** | Real-ESRGAN 生成式重建，CPU 上极慢；故 CPU 环境默认关闭 |
| 多目标追踪（§6.3） | ❌ CPU | 卡尔曼滤波 + IoU/匈牙利匹配，轻量 CPU 算法 |
| 多帧投票 / 规则纠错（§6.5/6.6） | ❌ CPU | 纯 Python 逻辑（对付模糊的**主力反而不吃 GPU**） |
| VLM 兜底（§6.8） | ❌ **云端 API** | ⚠️ 调 Claude/GPT/GLM 接口，**不占本地 GPU**，勿混淆 |
| 抽帧 / 回写编码（§6.1/§7） | 🔶 可选 | 默认 CPU 编解码；可选 NVENC/NVDEC 硬件编解码提速（优化项，非强依赖） |

> **一句话**：GPU = 给「检测 + OCR（+ 按需超分）」这几个**逐帧要跑上千次的神经网络**加速；其余环节要么 CPU、要么云端。

### 13.2 算力预算与降级路径

- **显存**：检测 + OCR（HyperLPR3 / onnxruntime-gpu）是轻量模型，占用小；**超分（Real-ESRGAN）是显存大头**，需求明显跳升。据此排机器——只做检测+OCR，入门级卡即可；要常态开超分，备中高端卡（具体数值待 P5.1/P5.4 实测，不预先假精确）。
- **哪些必须 GPU、哪些能省**：
  - *必须 GPU 才划算*：**检测、OCR**（逐帧密集，CPU 会成瓶颈）。
  - *能省 / 可退化*：超分（仅低置信 track 触发，可整体关）；硬件编解码（可退 CPU）；VLM 兜底（本就不占本地 GPU）。
  - *与 GPU 无关*：追踪、投票、规则纠错（纯 CPU，换 GPU 也不会更快）。
- **时延（GPU）**：几分钟视频、5fps 密采，检测+OCR 通常分钟级；回写逐帧编码是另一大头（与时长×分辨率线性相关，可选 NVENC 提速）。
- **时延（CPU 保底，§10）**：显著变慢，靠降 `PLATE_SAMPLE_FPS` + 关超分控制，并在 caveats 声明局限。

### 13.3 成本（API token）

- **不烧 token 的部分**：检测/追踪/OCR/融合/回写全是本地计算，**零 API 成本**。
- **唯一烧 token**：§6.8 VLM 兜底，仅对少数疑难 track、每 track 2–3 张小图，支持 caching，成本可控。契合明察「先用便宜的本地兜大头、VLM 只补疑难」的省钱哲学。
- **护栏**：`MAX_FRAMES_HARD`、`PLATE_ANNOTATE_MAX_SEC`、分段处理，防长视频撑爆内存/磁盘。

## 14. 测试策略（延续「mock、不烧 token、不需网络」）

现有 9 项测试全绿且全 mock。PLATE 测试同样**不真调 CV 模型、不烧 token**：

- **mock 检测/OCR**：`plate/` 门面在测试中被 monkeypatch 为返回构造的 `PlateBox`/`PlateDetection` 序列（如同现有测试 mock `llm` 门面）。
- **§6.5 投票单测**（纯函数，重点）：构造多帧含噪读数（如 `京A1234S`/`京AI234S`/`京A1234S`），断言投票 + 规则纠错定案为正确车牌。这是对付模糊的核心逻辑，必须有确定性单测。
- **§6.6 规则层单测**：0/O、8/B 等混淆在各字符位的纠正。
- **G1/G6 穿透测试**：分段抽帧 `-ss` 回加 t0 后时间正确；bbox 坐标系还原正确。
- **诚实否定测试**（守 NFR-1）：mock「检出但全模糊」→ 断言走否定 + caveats 非空（对齐现有 `test_locate_not_found`）。
- **降级测试**：mock `[plate]` 未安装 → 断言返回可读提示 Answer 而非抛异常。
- **回写**：mock ffmpeg（校验命令拼装与坐标还原），不实际编码。

## 15. 分阶段实现路线（P 系列里程碑）

沿用主设计文档的 P 编号风格，PLATE 记为 **P5**：

| 阶段 | 内容 | 交付 |
|---|---|---|
| **P5.0 骨架** | `Intent.PLATE` + intent/planner/dispatch 接入 + `from_plate` 否定桩 + `[plate]` extra + 降级路径 + mock 测试 | CLI 认得 PLATE，未装 extra 给可读提示，测试绿 |
| **P5.1 检测+追踪+OCR** | `plate/detect+track+ocr` 接 HyperLPR3 + ByteTrack；出 `PlateTrack`（未融合） | 能逐帧检出、跨帧关联、逐帧读数 |
| **P5.2 多帧融合（对付模糊·核心）** | `plate/fuse` 投票 + 规则约束；确定性单测 | 模糊车牌准确率显著提升，定案一个号 |
| **P5.3 高亮回写** | `annotate.py` bbox 插值 + 中文标签 + ffmpeg 合成 + 音轨 | **产出 `annotated.mp4`，标注跟随移动（达成交付目标）** |
| **P5.4 疑难增强** | 低置信触发超分（§6.7）+ VLM 兜底（§6.8）+ 透视矫正（§6.9） | 疑难/斜视角/极模糊车牌兜底 |
| **P5.5 前后端** | IntentPicker/ResultView 车牌 UI + 标注视频播放端点 + schema 同步 | Web 端可选车牌意图、看标注视频、看轨迹列表 |

**建议先做 P5.0 → P5.3 打通「能出标注视频」的最短闭环**（满足需求方交付目标），再迭代 P5.4/P5.5 提精度与体验。

## 16. 风险与开放问题

| 项 | 说明 | 处置 |
|---|---|---|
| HyperLPR3 版本/权重可用性 | pip 版本、权重下载源、GPU 兼容需实测 | P5.1 先验证；不行则切 YOLO-plate + PaddleOCR（门面隔离，不改 analyzer） |
| `onnxruntime-gpu` 与本机 CUDA 匹配 | 版本不匹配会 import 失败 | 降级 CPU + 文档标注 CUDA/cuDNN 要求 |
| 回写长视频耗时/体积 | 逐帧编码是大头 | `PLATE_ANNOTATE_MAX_SEC` 护栏 + 只标命中段 + 降帧率 |
| ByteTrack 依赖来源 | 独立实现 or 随 ultralytics（引 torch，重） | 优先轻量独立实现，避免 torch 进 [plate] 主 extra |
| 超分体积大 | realesrgan/basicsr 体积与依赖重 | 再拆 `[plate-superres]` extra，默认不装 |
| 中文标签字体 | 回写汉字需字体文件 | 随包附带开源字体 or 探测系统字体，缺失降级为拼音/省份代码 |
| 双层黄牌 / 新能源 8 位 | 长度与结构不同 | §6.5 按长度众数归组 + §6.6 结构分支 |
| 隐私合规 | 明文车牌输出 | §1.3 授权声明 + 产物不入库；预留反向「打码」R 选项 |

---

> **总结**：PLATE 在明察既有五层编排上**同构接入**（改 6 个现有文件 + 新增 `plate/` 子包与 `annotate.py`），核心价值是引入 **G6 空间-时序追踪地基**，让明察从「时间维度的语义分析」扩展到「空间维度的目标追踪 + 视频回写」。对付车牌模糊走「多帧投票（主）→ 规则约束 → 超分 → VLM 兜底」组合拳，其中 VLM 兜底是明察相对纯 CV 方案的差异化。全程遵守明察的诚实优先、永不崩、config 驱动、可选 extra 隔离、mock 测试不烧 token 五条铁律。
