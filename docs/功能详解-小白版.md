# 明察 MingCha —— 六大功能详解（小白版）

> 这份文档讲清楚界面上**「意图」那一排按钮**（下图红框）到底是什么、每个功能背后是怎么"想"出答案的。
> 全程尽量不用术语；遇到绕的地方都会打个生活里的比方。看完你就能明白：明察不是"魔法"，而是一套很聪明的"笨办法"。

```
┌─ 意图 ─────────────────────────────────────────────────────────────┐
│  [自动判断]   [理解摘要]   [定位]   [审核]   [以图搜]              │
│  [车牌/车辆]                                                        │
└────────────────────────────────────────────────────────────────────┘
```

对应界面上的六个按钮：

| 按钮 | 一句话说明 | 什么时候用它 |
|---|---|---|
| **自动判断** | 看你怎么问，自动挑下面五个里最合适的（推荐） | 不想动脑，直接打字提问 |
| **理解摘要** | 把视频讲了啥，结构化总结出来 | "这视频在讲什么？" |
| **定位** | 找某个东西**最早**在第几秒出现 | "红色的车最早出现在什么时间？" |
| **审核** | 排查有没有敏感/违规内容，宁可多报不漏报 | "有没有暴力、血腥画面？" |
| **以图搜** | 你给一张图，在视频里帮你找出现的位置 | 附张照片 +「这个人出现在哪？」 |
| **车牌/车辆** | 追踪车辆、画框跟随，识别框里的车牌号 | "识别视频里的车牌" |

## 一、先搞懂一件事：明察是怎么"看"视频的

计算机其实**不会直接看视频**。视频对它来说就是一大坨数据。明察的第一步，永远是把视频"拆成一张张图片"——就像把一部电影拆成一本**连环画**。

> 🎬 **核心类比**：明察干的事，本质是"**把电影拆成连环画，再逐页看图说话**"。
> 后面六个功能，区别只在于"怎么看这些连环画"——是通读一遍写读后感（摘要），还是一页页翻找某个东西（定位），还是拿放大镜逐格排查（审核）。

拆图的时候，有一个东西**绝对不能丢**：**每张图是视频第几秒的**。

这就是明察最看重的"**时间戳地基**"。你可以理解为：连环画的每一页角上，都盖了一个精确到毫秒的**时间图章**（比如 `00:01:23.456`）。有了这个图章，明察才能回答"这东西出现在**第几秒**"，而不是含糊地说"出现在某一页"。

## 二、所有功能共用的地基：预处理流水线

不管你点哪个功能，明察拿到视频后都会先跑一条统一的"**预处理流水线**"，把视频变成好分析的素材。这条流水线一共 8 步，你可以想象成一条**工厂流水线**：

```
 原视频
   │
   ├─①下载/读取视频 ────────────► 拿到 video.mp4
   │
   ├─②抽帧（拆成图片）───────────► frame_001.jpg, frame_002.jpg ...
   │      同时给每张图盖上"时间图章"
   │
   ├─③解析时间戳 ───────────────► 记下"第3张图 = 第12.5秒"
   │
   ├─④去重（扔掉几乎一样的图）───► 100张里可能只留30张有用的
   │
   ├─⑤语音转写（把说的话变文字）─► transcript.txt（有字幕/装了whisper才有）
   │
   ├─⑥抽音频（可选）────────────► audio.wav（审核时用来听尖叫等声音）
   │
   ├─⑦写时间戳表 ───────────────► timestamps.json（图片↔秒 的对照表）
   │
   └─⑧拼图（把多张图拼成一张大图）► grid_01.jpg（3×3宫格，每格标时间）
```

下面挑三个小白最该懂的步骤，讲讲它们**为什么这么做**。

### 抽帧：为什么不是每一帧都要？

一个视频一秒可能有 25~30 帧（张画面）。全都拿去分析，又慢又贵。所以明察**抽稀**——隔几帧取一张。而且它更聪明：**画面切换的地方多取几张**（因为那里信息量大），画面不动的地方少取。

> 🍜 **类比**：看一部电视剧，你不会一帧一帧看，而是"剧情有变化时多留意，长镜头空镜快进过去"。抽帧就是这个道理。

### 去重：把"几乎一样的图"扔掉

抽出来的图里，常有很多长得几乎一模一样（比如一个人在说话，背景 5 秒没变）。明察会把它们**去重**，只留有代表性的。

它怎么判断两张图"像不像"？办法特别朴素：把两张图都**缩成 16×16 的小马赛克**（256 个色块），然后一个格子一个格子比颜色。如果大部分格子颜色都差不多，就认为是"重复的"，扔掉。

> 🖼️ **类比**：把两张照片都眯着眼看成一小团色块，如果两团色块长得差不多，就当成同一张。简单，但很有效，而且**不费一分钱、不用联网**。

⚠️ 注意：**去重时时间图章会一起跟着搬家**。这一点是明察特意做的——普通工具去重后重命名，时间信息就丢了，明察坚持让"第几秒"这个信息穿过整条流水线不掉队。

### 拼图：把 9 张小图拼成 1 张大图

明察会把关键帧 **3×3 拼成一张"九宫格大图"**（叫 contact sheet / 拼图），每个格子左上角标着它的时间（`00:00:12.500`）。

为什么要拼？因为把 9 张图**一次性**发给 AI 模型，比发 9 次**省钱、省时间**，而且模型能一眼看到前后画面的联系。

> 📇 **类比**：就像冲洗照片时的"**印样小样**"——一张纸上密密麻麻印了一卷胶卷的缩略图，方便一眼扫过全局。

📌 **一句话记住这一节**：明察先把视频变成"**一叠盖了时间章的图片 + 一份台词文稿**"，后面六个功能都是在这堆素材上做文章。

> 想深入的看代码：`preprocess.py`（流水线）、`timestamps.py`（时间图章）、`similarity.py`（16×16 比图）。

## 三、功能① 自动判断（意图识别）

### 它解决什么问题？

你在输入框打一句话，比如"红色的车最早出现在什么时间"。明察得先**搞懂你到底想干嘛**——是要总结？要定位？还是要审核？搞懂了才能派对应的"专家"去干活。这一步叫**意图识别**。

界面上选"自动判断"，就是把这一步交给明察自己拿主意。

### 实现原理：先用"关键词规则"，不行才请 AI

明察判断意图，用的是**两道关卡**，能省钱就省钱：

```
你的问题
   │
   ▼
第一关：关键词规则（几乎零成本，一瞬间出结果）
   ├─ 你上传了参考图？        → 直接判为「以图搜」
   ├─ 出现"车牌/牌照/车号"？   → 判为「车牌/车辆」
   ├─ 出现"最早/第几秒/何时"？ → 判为「定位」
   ├─ 出现"有没有/是否/审核"？ → 判为「审核」
   ├─ 出现"总结/讲了什么"？    → 判为「理解摘要」
   │
   └─ 一个关键词都没命中？
          │
          ▼
   第二关：请 AI 帮忙分类（花一点点钱，但更聪明）
          │
          └─ 万一 AI 也出错/没网/没Key？
                 │
                 ▼
          兜底：当成「理解摘要」处理（保证永远不崩）
```

> 💡 **为什么这么设计？**
> - **省钱**：大部分问题一眼就能看出意图，用关键词秒判，何必花钱问 AI？
> - **稳**：就算 AI 服务挂了、没联网、Key 填错，也有"兜底为摘要"这条后路，**程序永远不会崩**。
> - **顺手**：有个小心机——"车牌"这个词的判断**排在"审核"前面**。因为"识别车牌"里的"识别"本来会被"审核"抢走，特意让车牌优先，避免误判。

### 举几个例子

| 你输入 | 明察判断 | 靠什么判的 |
|---|---|---|
| `总结一下这视频` | 理解摘要 | 命中关键词"总结" |
| `刀最早出现在几分几秒` | 定位 | 命中"最早""几分几秒" |
| `有没有血腥画面` | 审核 | 命中"有没有" |
| （附了张人脸图）`他在哪出现` | 以图搜 | 有参考图，强信号 |
| `帮我看看车牌号` | 车牌/车辆 | 命中"车牌" |
| `这个视频里发生了啥冲突` | （关键词没命中）→ 请 AI 判 | AI 分析语义 |

> 想深入的看代码：`intent.py`（两道关卡）、`prompts.py` 里的 `CLASSIFY_SYSTEM`（给 AI 的分类说明书）。

## 四、功能② 理解摘要

### 它解决什么问题？

"这视频到底讲了啥？"——一句话，帮你把整个视频**读一遍并写出结构化的读后感**：主题是什么、分了几段、有哪些要点。

### 实现原理：这是最简单的一个——看图说话，一次搞定

摘要不需要"逐帧找东西"，所以它是六个功能里最简单的：

```
   拼图（九宫格大图，带时间戳）
        +                     ──►  发给 AI 模型  ──►  结构化摘要
   语音转写文稿（台词）              （一次调用）        ├─ 主题（一句话）
                                                        ├─ 分段脉络（按时间）
                                                        └─ 关键要点
```

就三步：

1. **备料**：把预处理产出的"九宫格拼图"+"台词文稿"拿出来。（没拼图就退而用单张关键帧；没台词就告诉 AI"只看画面")
2. **发给 AI 一次**：让它同时看图和读台词。
3. **收结构化结果**：AI 不是随便写一段，而是按固定格子填——**主题 / 分段 / 要点 / 完整摘要**，方便界面分区漂亮地展示。

> 🗣️ **类比**：就像请一个助理，把一部纪录片的"印样小样 + 台词本"递给他，让他半小时内交一份**提纲式读后感**。图上带着时间戳，所以他还能告诉你"第 3 分钟讲了 XX"。

一个小细节：摘要这类"理解题"没有标准答案，不像"定位"能验证对错，所以明察给它一个**固定的中高可信度（0.8）**，而不是硬编一个精确分数。

> 想深入的看代码：`analyzer/summary.py`、`prompts.py` 里的 `SUMMARY_SYSTEM`。

## 五、功能③ 定位

### 它解决什么问题？

"**那把刀最早在第几秒出现？**"——不是问有没有，而是要一个**精确到秒**的时间点，而且要"**最早**"的那次。

### 实现原理：两阶段，"先粗后细"

如果每 0.3 秒就仔细看一帧，一个 10 分钟视频要看 2000 帧，太慢太贵。明察用**两阶段**，像查字典一样"先翻到大概位置，再逐字找"：

```
阶段一【粗扫】：稀疏地快速过一遍，先圈定大概位置
────────────────────────────────────────────────
 帧1   帧2   帧3   帧4   帧5   帧6 ...
  ❌    ❌    ❌   ✅命中!  ...      ← 逐帧问AI"这帧有刀吗？"
                  │
                  │ 找到最早命中的大概在第 40 秒
                  ▼
阶段二【精扫】：只在这一小段里，密集地找"第一次"
────────────────────────────────────────────────
 只在 第32秒~第48秒 这 16 秒内，每 0.3 秒抽一帧，重新逐帧问
 32.0  32.3  32.6 ... 39.8  40.1 ...
                        ✅ 最早在 39.8 秒真正出现！
                        ▼
              回答："目标最早出现在 00:00:39.800"
```

- **阶段一（粗扫）**：拿预处理好的稀疏关键帧，**一张张问 AI**："这帧里有没有刀？"AI 对每帧回一个"有/没有 + 有多大把握"。取**最早**那个有把握的命中，锁定大概位置。
- **阶段二（精扫）**：在命中点**前后各 8 秒**这一小段里，**重新用高密度（每 0.3 秒一帧）抽帧**，再逐帧问一遍，把时间点**磨到秒级精度**。

> 🔍 **类比**：在一本书里找某句话。你不会一个字一个字从头读，而是**先翻目录/快速翻页**定位到大概哪一章（粗扫），**再在那一页仔细逐行看**（精扫）。

### 关键点：找不到就老实说找不到

如果扫完确实没有，明察**不会瞎编**，而是明确回答"在抽取的关键帧里没发现"，并**主动声明局限**："我是按采样看的，特别短暂的一闪可能没抽到"。这份"诚实"是明察的铁律之一（见[第九节](#九贯穿始终的三条铁律)）。

> 想深入的看代码：`analyzer/locate.py`、`rescan.py`（精扫时的密集抽帧）。

## 六、功能④ 审核

### 它解决什么问题？

"这视频**有没有**暴力、血腥、色情等敏感内容？"——用于内容安全排查。这里的核心诉求和"定位"相反：**宁可错报，也绝不漏报**。

### 实现原理：地毯式排查 + 把命中点连成"时间段"

因为审核**怕漏**，所以明察在这个模式下会：

1. **关掉去重、加密采样**：普通模式会扔掉相似帧，但审核**一帧都不敢扔**（万一漏掉的那帧正好有问题）。所以它分段密集抽帧，尽量多看。
2. **逐帧地毯式排查**：每一帧都问 AI"这帧有没有敏感内容？"，而且**判定的门槛调得很低**——只要有点像，就先记为"命中"。

```
      逐帧排查（门槛调低，宁可错报）
   帧: ✅ ✅ ❌ ❌ ✅ ✅ ✅ ❌ ❌ ❌ ✅ ✅
       └─┬─┘        └──┬──┘          └┬┘
         │             │              │
         ▼             ▼              ▼
   把"挨得近"的命中点合并成时间段（间隔≤3秒就并起来）
   00:01~00:03   00:08~00:12    00:20~00:21
```

3. **把零散命中点合并成区间**：单个命中点意义不大，明察把**挨得近（间隔 ≤ 3 秒）的命中点合并成"时间段"**，比如"第 8 秒到第 12 秒疑似有问题"，方便人去复核。

> 🚨 **为什么"宁可错报"？** 审核的场景里，漏掉一个违规画面的代价，远大于误报几个。所以明察故意把门槛调低、召回拉满（这叫"**高召回**"），最后一定**交给人来最终拍板**。它自己只做"初筛"，绝不替人下定论。

### 关键点：合规红线 + 未成年人特别标注

- AI 在审核时**只回答"有没有"，绝不描述、复述任何露骨细节**（这是合规要求，写在给 AI 的指令里）。
- 一旦**疑似涉及未成年人**，或者 AI **拒绝回答**某帧，明察会**特别标注"建议高优先级人工复核"**，把最敏感的部分顶到人工面前。

> 想深入的看代码：`analyzer/moderate.py`、`prompts.py` 里的 `moderate_system`（含合规约束）。

## 七、功能⑤ 以图搜

### 它解决什么问题？

你手上有**一张图**（某个人、某辆车、某个物件），想知道"**它在视频里的哪个时间出现过**"。文字描述不清的东西，用图来找最直接。

### 实现原理：三级漏斗，"由粗到细"层层筛

如果拿你的图和视频每一帧都让 AI 仔细比对，太贵了。明察用**三级漏斗**，先用便宜手段筛掉大部分，只把少数"可疑"的送去精比对：

```
   全部关键帧（比如 200 张）
        │
   ┌────▼────────────────────────────────────┐
   │ 级① 像素预筛（免费、不联网、飞快）        │
   │ 把你的图和每帧缩成小马赛克比颜色，        │
   │ 只留最像的前 15 张                        │
   └────┬────────────────────────────────────┘
        │  200张 → 15张
   ┌────▼────────────────────────────────────┐
   │ 级② 语义确认（请 AI 逐张仔细比对）        │
   │ 把"参考图 + 候选帧"成对发给 AI，          │
   │ 判：同一个 / 只是像 / 不是               │
   └────┬────────────────────────────────────┘
        │  15张 → 找到最早的真命中
   ┌────▼────────────────────────────────────┐
   │ 级③ 精扫（同"定位"，磨到秒级）            │
   └──────────────────────────────────────────┘
```

- **级① 像素预筛**：用[第二节](#二所有功能共用的地基预处理流水线)讲过的"16×16 比色块"办法，把你的图和每一帧快速比一遍，**只挑出最像的前 15 张**。这一步不花钱、不联网。
- **级② 语义确认**：把"你的参考图 + 一张候选帧"**成对**发给 AI，让它认真判断：这是**同一个**东西，还是**只是长得像**，还是**根本不是**。
- **级③ 精扫**：找到最早的真命中后，再像"定位"一样在附近密集抽帧，把时间磨准。

### 关键点：分清"同一个" vs "长得像"

这是"以图搜"最讲究的地方。明察会**严格区分**：

- **same（同一个体）**：确实是你图里那个东西本身。
- **similar（同类外观）**：只是同款、同色、长得像，**不能确定是同一个**。

比如你给一张"红色轿车"的图，视频里出现另一辆一模一样的红色轿车——明察会诚实地说"找到**外观相似**的，但不能确定是同一辆"，而不会拍胸脯说"就是它"。**相似 ≠ 同一**，这个分寸它拿捏得很清楚。

> 💰 小彩蛋：你的参考图在每次比对里都要用到，明察会把它标记为"可缓存"，支持该功能的模型会自动省下重复上传的钱。

> 想深入的看代码：`analyzer/visual.py`、`similarity.py`（像素预筛）、`prompts.py` 里的 `visual_system`。

## 八、功能⑥ 车牌/车辆（重点详解）

这是明察**最新、也最复杂**的功能，值得单独细讲。它要做的事，用界面上的话说就是：**"追踪车辆并高亮跟随，框内识别车牌"**——最终产出一段**画了框、框会跟着车走、框上写着车牌号**的新视频。

### 先解决一个现实难题：为什么要分"两种模式"？

现实中很多监控视频是**高处俯拍车流**，车牌只有**十几个像素**那么小——糊得连人都看不清，专门认车牌的模型根本"抓不住"它，导致每一帧都断片、没法连续追踪。

明察的应对：**认不清小车牌，那就先认"整辆车"**。车是大目标，好认、好跟。于是有了两种模式：

| 模式 | 先找什么 | 适合场景 | 车牌怎么办 |
|---|---|---|---|
| **车辆模式**（默认） | 用 YOLO 找**整辆车** | 俯拍车流、车牌很小 | 在每辆车的框**里面**再找车牌，找到就标，找不到就只框住车跟随 |
| **车牌模式** | 直接找**车牌** | 车牌够大够清楚 | 直接识别 |

> 🚗 **类比**：远处马路上一辆车，你先一眼认出"那是辆车"（容易），走近了才看清车牌（难）。明察就是这个顺序——**先锁定车，再努力看牌**。

### 完整流水线：一辆车的车牌是怎么被"拼"出来的

车牌功能内部是一条 6 步流水线。我们跟着"一辆车"走一遍：

```
连续帧（5fps 密集抽帧，不去重——追踪需要连续画面）
   │
   ├─❶ 检测：每帧找出"车（或车牌）在画面哪个位置" ──► 一堆小框
   │
   ├─❷ 追踪：把不同帧里"同一辆车"的框串成一条线 ──► 一条车的"轨迹"
   │
   ├─❸ 识别(OCR)：把每帧框里的车牌图，读成文字 ──► 每帧一个（可能带错的）读数
   │
   ├─❹ 多帧投票融合：几十个模糊读数投票，拼出正确车牌号 ──► 定案车牌号 ★核心★
   │
   ├─❺ 疑难兜底：还是没把握的少数车牌，请 AI 专家再看 ──► 纠正
   │
   └─❻ 高亮回写：把框和车牌号画回视频，框跟着车走 ──► annotated.mp4
```

下面逐步拆解，重点在**第❷步追踪**和**第❹步投票**——这两步是明察的精髓。

#### ❶ 检测：先把车/车牌圈出来

对每一帧，用检测模型圈出目标的位置（一个个方框）。车辆模式用 **YOLO**（通用物体检测，认车很擅长），车牌模式用 **HyperLPR3**（国产、专门认中国车牌，蓝牌黄牌绿牌都认）。

这一步只管"**东西在哪**"，还不管"车牌号是啥"。

> 🔧 一个工程小细节：检测和识别共用**同一个模型实例**。因为这个模型第一次运行会下载几十兆的权重文件，如果开两份就会在 Windows 上"打架"（两个程序同时解压一个文件会报错）。共用一份，既避免打架又省内存。

#### ❷ 追踪：把"同一辆车"在不同帧里的框串起来 —— 这是关键

检测只告诉你"第 10 帧这里有辆车、第 11 帧那里有辆车"，但**它俩是不是同一辆车**？追踪就是干这个的：把连续帧里属于同一辆车的框，**串成一条轨迹**。

它怎么判断"这两帧的框是同一辆车"？靠两个朴素的直觉：

1. **位置重叠**：同一辆车在相邻两帧里，位置不会突变，两个框应该**大面积重叠**。（专业叫 IoU，"交并比"，其实就是"两个框重叠了多少"）
2. **匀速预测**：车在动，明察会**根据它前几帧的速度，预测它下一帧大概在哪**，然后去那个位置附近找。

```
   第10帧        第11帧        第12帧
   ┌──┐         ┌──┐          （这帧没检到，被树挡住了）
   │车│  ──►    │车│   ──►    预测它应该到这 ┌──┐  ──► 第13帧续上
   └──┘         └──┘          （短暂消失也不丢） └··┘
   位置A         位置B
   └──── 重叠+匀速 → 判定是同一辆车，串成一条轨迹 ────┘
```

还有个贴心设计：**短暂被挡住也不算丢**。比如车被路牌挡了几帧，明察会**容忍这段"断片"**（最多容忍 15 帧），等它再出现就接着串上，而不是当成两辆新车。

> 🧵 **类比**：看一群人走过，你能认出"刚才那个红衣服的人，现在走到那边了"——靠的就是"位置连续 + 速度合理"，哪怕他中间被别人挡了一下。追踪就是让计算机也会这个。

**为什么追踪这么重要？** 因为它把"同一辆车的几十帧读数"**攒到了一起**，这才有了下一步"投票"的本钱。没有追踪，每帧都是孤立的、模糊的读数，谁也纠正不了谁。

#### ❸ 识别（OCR）：逐帧读牌

对轨迹上每一帧的车牌小图，**裁剪出来单独放大再读一次**（比在整帧里读更准）。得到的是**每帧一个读数**——注意，这些读数**单看都可能是错的**（因为每帧都有点糊）。比如同一辆车：

```
   第10帧读成：京A·1234
   第11帧读成：京A·12E4   ← 3 看成了 E
   第12帧读成：京4·1234   ← A 看成了 4
   第13帧读成：京A·1234
   ...
```

#### ❹ 多帧投票融合：把一堆"模糊错读"拼成"正确车牌" —— 这是明察的精髓 ★

单帧读不准没关系，明察有杀手锏：**让几十帧一起投票**。

车在动，同一个车牌经过几十帧，**每帧糊的地方都不一样**。那就**按车牌的每一位分别投票**，每一位都取"得票最高"的那个字：

```
        第1位  第2位  第3位  第4位  第5位  第6位  第7位
 第10帧:  京     A     ·     1     2     3     4
 第11帧:  京     A     ·     1     2     E     4    ← 第6位读错
 第12帧:  京     4     ·     1     2     3     4    ← 第2位读错
 第13帧:  京     A     ·     1     2     3     4
  ...    ...   ...   ...   ...   ...   ...   ...
 ─────────────────────────────────────────────────
 投票:   京    A(多)  ·     1     2    3(多)   4
 结果:   京     A     ·     1     2     3     4   ✅ 拼对了！
```

第 2 位大多数帧读的是"A"，个别读成"4"——**少数服从多数**，定为 A。第 6 位同理定为 3。这样**投票拼出的车牌号，比任何单独一帧都准**。而且这几乎是**白捡的便宜**：追踪已经把这些帧攒好了，投票只是纯计算，不花一分钱。

投票时还有两个精细设计：

- **谁读得清，谁的票更重**：不是简单一帧一票，而是按"这帧读得有多清楚"加权。看得清的帧，票更值钱。
- **短板决定整体信心**（木桶效应）：整个车牌的可信度，取"**最没把握的那一位**"。只要有一位吃不准，整个车牌号就不敢标高信心。这是"诚实"的体现。

投完票，再用**中国车牌的规则**兜一道底，进一步纠错：

- 车牌**第 2 位一定是字母**（地区代码），如果读成了数字，按长得像的规则纠回去（比如 `0→O`、`1→I`、`8→B`）。
- 车牌**从第 3 位起不会用字母 I 和 O**（国标规定，怕和数字 1、0 混），所以读到 I 就纠成 1，读到 O 就纠成 0。
- **过滤明显是噪声的误读**：比如俯拍小目标常把画面误读成"京E苏211111"这种（除第一位外又冒出个汉字"苏"，不符合车牌结构），明察会识别出"这不是合法车牌格式"，**大幅降低它的可信度并标注"疑似误识别"**，不让它冒充真车牌来误导你。

> 🗳️ **类比**：一群人隔着毛玻璃看同一块车牌，每人抄下来的都有点出入。你把大家的答案**按位对齐一比**，"多数人这一位都写 A"，那这一位基本就是 A。人多，眼就亮。

#### ❺ 疑难兜底：少数还没把握的，请 AI 专家再看一眼

投票 + 规则之后，**绝大多数车牌已经定案**。只有**少数几个仍然没把握**的（可信度还是偏低），明察才会启动"兜底"：

1. 挑出这辆车**最清晰的 2~3 张**车牌裁剪图；
2. 可选地做**透视矫正**（把拍歪的车牌摆正）和**增强/超分**（提清晰度）；
3. 连同"投票的候选答案"一起发给 **AI 大模型（VLM）**，让它当"车牌专家"，按中国车牌规则给出最可能的读数和逐位理由。

> 💸 **这是整个车牌功能里唯一"花钱"（调用 AI）的环节**，而且只对少数疑难车牌、每辆车就 2~3 张小图，成本很低。绝大多数车牌靠前面免费的投票就搞定了。

如果 AI 也拒答、或没配 Key，就**保留投票的结果**，绝不因此崩溃。

#### ❻ 高亮回写：把框和车牌号画回视频

最后一步，产出你能直接看的成果——一段**标注视频 `annotated.mp4`**：每辆车一个**彩色框**，框**跟着车移动**，框上贴着**最终定案的车牌号**。

几个讲究的地方：

- **框会平滑跟随**：检测只在部分帧有框，明察在两帧之间**做插值**（算出中间时刻框该在哪），让框顺滑地跟着车走，而不是一跳一跳的。
- **标签不闪烁**：框上的车牌号用的是**最终定案的那个号**（不是每帧抖动的原始读数），所以你看到的车牌号是稳定的，不会一帧一个样地乱跳。
- **坐标要还原**：检测是在缩小过的画面（宽 1280）上做的，画框却要画回**原始分辨率**的视频，明察会**按比例把框放大还原**，否则框就会画歪、对不上车。
- **颜色跟牌走**：蓝牌画蓝框、绿牌（新能源）画绿框，一目了然。
- **保留原声**：标注视频会**保留原视频的声音**。

> 🎥 **类比**：就像电视里追拍嫌疑车辆时，画面上那个**一直套着车、还标着车牌号的方框**。明察自动帮你把这个效果做出来。

### 车牌功能总结

```
 认不清小车牌 → 先追整辆车（大目标好追）
                     │
 单帧读不准    → 追踪攒够几十帧 → 按位投票（人多眼亮，免费）
                     │
 投票没把握    → 规则纠错 + 少数疑难请AI专家（唯一花钱处）
                     │
 最终成果      → 画框跟随 + 车牌号标签的新视频
```

> 想深入的看代码（都在 `src/mingcha/plate/` 目录）：
> `detect.py`（车牌检测）、`vehicle.py`（车辆检测）、`track.py`（追踪）、`ocr.py`（读牌）、`fuse.py`（投票+规则，**精华**）、`vlm_fallback.py`（AI兜底）、`__init__.py`（总编排）；画框回写在 `annotate.py`。

## 九、贯穿始终的三条铁律

明察这六个功能，无论多复杂，都守着三条底线。理解了它们，你就懂了明察的"性格"：

### 铁律一：诚实优先 —— 不知道就说不知道

明察**绝不为了给答案而瞎编**。找不到就明说"没找到"，并且**必须声明局限**："我是按采样看的，特别短暂的画面可能没抽到"。车牌太糊读不出，就说"检出了车牌但太模糊、无法可靠识别"，而不是硬猜一个号骗你。

> 这一点在敏感场景（审核、车牌）尤其重要——**宁可说"要人工复核"，也不给一个可能误导你的假答案**。

### 铁律二：永不崩溃 —— 任何一步出错都有后路

- AI 分类失败？→ 兜底当成"摘要"。
- 某一帧 AI 拒绝回答？→ 跳过这帧，不影响其他帧。
- 车牌依赖没装好？→ 提示你怎么装，其他功能照常用。
- 缓存文件坏了？→ 当没缓存，重新算一遍。

明察的每一个环节都想好了"万一这步挂了怎么办"，所以它**很难被单点故障搞崩**。

### 铁律三：换模型只改配置，不用改代码

明察不绑死任何一家 AI（Claude / GPT / GLM 都能用）。想换模型，通常只要**改一行配置**，代码一行不用动。这让它很灵活，哪家便宜好用就用哪家。

## 十、附录：跑完之后产出了哪些文件

一次分析跑完，明察会在输出目录（默认 `mingcha-out/`）留下这些东西，方便你核对：

| 文件/目录 | 是什么 |
|---|---|
| `answer.json` | **最终答案**（结构化），界面就是读它来展示的 |
| `frames/` | 抽出并去重后的**关键帧图片** |
| `grids/` | 拼好的**九宫格大图**（摘要用） |
| `timestamps.json` | **图片↔第几秒** 的对照表（时间图章总表） |
| `transcript.txt` | **语音转写**的台词（如果有） |
| `annotated.mp4` | **车牌/车辆**功能专属：画了框、带车牌号的**标注视频** |
| `query_image.jpg` | 以图搜时，你上传的**参考图**的归档 |
| `.mingcha_cache.json` | 缓存指纹。同样的视频+同样的问题，下次**直接复用结果**，不重复烧钱 |

> 💡 **缓存机制**：明察会记住"这个视频 + 这个问题 + 这套参数"的组合。下次你问一模一样的，它直接把上次的结果给你，**又快又省**。想强制重算，加 `--no-cache`。

## 一页纸速记

| 功能 | 核心套路 | 一句话记忆 |
|---|---|---|
| **自动判断** | 关键词秒判 → 不行请AI → 兜底摘要 | 先用笨办法，省钱又稳 |
| **理解摘要** | 拼图+台词 → AI看一次 → 结构化读后感 | 看图说话，一次搞定 |
| **定位** | 粗扫圈范围 → 精扫磨到秒 | 先翻页定位，再逐行细看 |
| **审核** | 密采不去重 → 低门槛地毯排查 → 合并成时间段 | 宁可错报不漏报，交人工拍板 |
| **以图搜** | 像素预筛 → AI语义确认 → 精扫 | 三级漏斗，分清"同一个"和"像" |
| **车牌/车辆** | 追车→攒帧→按位投票→AI兜底→画框回写 | 人多眼亮，几十帧投票拼出车牌 |

> 全流程一句话：**把视频拆成带时间戳的图片，再针对你的问题，用最省的办法看这些图、给出带证据的答案。**
